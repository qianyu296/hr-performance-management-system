# 02 概要设计

## 1. 目的与范围

本文将《01-需求分析》中的业务目标转化为可实施的系统结构，明确应用边界、模块职责、关键流程和部署方式。系统一期采用前后端分离的模块化单体架构，面向 500 至 2,000 名员工规模的中小企业。

本文不定义数据库字段、接口参数或页面组件细节。它们分别由数据库设计、接口设计和前端设计文档约束。

## 2. 架构总览

```text
Vue 3 Web 管理端
        |
HTTPS / REST API
        |
Spring Boot 应用
  |- 认证与权限模块
  |- 组织人事模块
  |- 假勤模块
  |- 目标绩效模块
  |- 通知待办模块
  |- 报表模块
        |
MySQL / Redis / MinIO / Quartz
```

前端负责交互、展示和基础输入校验；后端负责认证、权限、业务规则、流程状态、事务和审计。MySQL 保存业务数据，Redis 保存登录会话、权限缓存、验证码及短期热点数据，MinIO 保存业务附件，Quartz 执行提醒、周期初始化和报表汇总等任务。

## 3. 后端分层

| 层级 | 职责 | 禁止事项 |
| --- | --- | --- |
| Controller | 参数接收、鉴权入口、响应封装 | 不写业务计算和跨模块事务 |
| Application Service | 编排用例、控制事务、调用领域服务 | 不直接拼接复杂 SQL |
| Domain | 状态机、审批、绩效计算、余额变化等规则 | 不依赖 HTTP 或页面模型 |
| Repository/Mapper | 数据读取与持久化 | 不承载流程判断 |
| Infrastructure | 缓存、文件、消息、任务、第三方接口 | 不向上泄漏具体实现 |

统一使用 DTO 处理接口输入输出，领域对象不直接暴露给前端。模块间优先通过应用服务调用和领域事件协作；一期不引入消息队列，异步任务先用数据库任务记录与 Quartz 执行。

## 4. 模块职责

| 模块 | 职责 | 主要依赖 |
| --- | --- | --- |
| 认证权限 | 登录、令牌、用户、角色、菜单、数据范围、审计 | Redis、组织人事 |
| 组织人事 | 组织、岗位、职级、员工档案、入转调离、履历 | 审批、附件、通知 |
| 假勤 | 规则、余额、请假、加班、调休、统计 | 审批、通知、报表 |
| 目标绩效 | 目标周期、指标方案、考核、校准、发布、申诉 | 组织人事、审批、通知 |
| 审批待办 | 审批模板、实例、节点、待办、站内消息 | 各业务模块 |
| 报表 | 聚合查询、导出、数据看板 | 全部业务模块 |
| 文件服务 | 附件上传、下载、预览授权、清理 | MinIO、权限 |

## 5. 核心设计原则

1. 人员、组织、岗位等主数据发生变化后，已办结业务使用快照或版本保留当时事实。
2. 所有受审批控制的业务必须通过状态机流转，禁止通过通用更新接口修改最终状态。
3. 权限校验在 Controller 与应用服务两层执行；任何查询、下载、导出都应执行数据范围限制。
4. 绩效结果、假期余额和离职办结为强一致性业务，必须使用数据库事务和乐观锁保护。
5. 所有关键动作写入审计日志；日志记录操作主体、对象、时间、结果和必要的变更摘要。
6. 配置一经被业务周期引用即冻结。修改时创建新版本，不能覆盖已生效的规则。

## 6. 关键流程编排

### 6.1 审批

业务模块创建业务单据后调用审批模块。审批模块根据模板、发起人组织和业务上下文生成实例与节点快照，同时创建当前审批人的待办。审批动作完成后，审批模块推进下一节点或回调业务模块更新单据状态；业务模块在最终通过时执行自己的生效逻辑。

### 6.2 绩效

HR 启动考核周期后，系统按适用范围生成考核任务并冻结方案版本。员工自评、主管评分、HR 校准和发布均在同一任务状态机内推进。分数计算由后端统一执行，发布后只允许通过申诉或归档更正形成受控变更。

### 6.3 提醒与任务

Quartz 定期扫描待办截止时间、试用期、合同到期、绩效节点和未更新目标。任务采用幂等键避免重复通知，通知写入站内信；未来可扩展企业微信或邮件适配器。

## 7. 部署建议

开发环境使用 Docker Compose 启动 MySQL、Redis、MinIO；应用和前端本地运行。测试与生产环境采用 Nginx 托管前端静态文件并反向代理 API，Spring Boot 以容器运行。所有环境配置通过环境变量或外部配置文件注入，禁止把数据库密码、存储密钥和 JWT 密钥提交到代码库。

## 8. 异常与可观测性

后端统一返回业务错误码和可读提示，未知异常不向前端暴露堆栈。请求携带 traceId，日志按 traceId 关联。记录接口耗时、慢查询、任务失败、登录失败和审批异常；管理端可查看任务执行与失败重试情况。

## 9. 可靠异步与一致性

业务事务内只完成业务数据、审计记录和 `sys_outbox_event` 写入。事务提交后，由 Quartz 工作节点领取 `sys_async_job` 或 outbox 事件，执行通知、导出、汇总、文件后处理和批量发布。不得在 Controller、事务回调或内存队列中直接发送通知。

1. 每个异步作业具有 `job_type + idempotency_key` 唯一键；消费者必须可重复执行，不得产生重复待办、重复余额流水或重复发布。
2. 工作节点以数据库条件更新领取任务，设置锁定超时；超时的 `RUNNING` 作业可由其他节点回收。失败按指数退避重试，超过最大次数进入 `DEAD` 并告警。
3. 绩效批量发布、导入和导出均创建可查询的批次任务。批次可部分成功，但周期处于 `PUBLISHING` 时不可关闭；管理员只能对失败子项重试，不能用手工 SQL 改状态。
4. 附件对象先进入隔离区。关联业务成功后由异步作业转为可用对象；恶意文件、超大文件、未知媒体类型和过期未关联对象必须被拒绝或清理。

## 10. 运行、备份与恢复

生产最小拓扑为 Nginx、至少两个无状态 Spring Boot 实例、MySQL 主实例、Redis、MinIO 和独立的定时任务执行实例。单实例部署只能用于开发和测试，不能作为生产高可用承诺。

| 对象 | 目标 | 实施要求 |
| --- | --- | --- |
| MySQL | RPO 24 小时，RTO 4 小时 | 每日全量备份、binlog 保留不少于 7 天、每季度恢复演练；迁移前备份并记录恢复点。 |
| MinIO 附件 | RPO 24 小时，RTO 4 小时 | 对象版本或跨桶备份；数据库文件元数据与对象清单联合校验。 |
| Redis | 可重建 | 仅缓存会话、权限和热点数据；重启后从 MySQL 重建，不保存唯一业务事实。 |
| 审计与日志 | 可追溯 | 应用日志集中采集；操作审计按企业策略长期归档，普通用户无修改权限。 |

所有密钥经环境变量或密钥管理服务注入，至少包含数据库、Redis、对象存储、JWT、加密主密钥。生产必须启用 HTTPS、HSTS、限流、对象存储私有桶和访问日志；密钥与管理员初始密码不得写入镜像、前端构建产物或 Git。

## 11. 后续扩展

当业务规模扩大时，可优先把报表导出、通知和文件处理拆为异步服务；复杂流程再接入 Flowable。对接考勤机、钉钉、企业微信、单点登录及薪酬系统时，通过适配器层隔离第三方协议。
