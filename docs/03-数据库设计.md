# 03 数据库设计

## 1. 设计目标

数据库采用 MySQL 8.0，使用 InnoDB 与 `utf8mb4` 字符集。设计重点是组织与员工主数据稳定、流程可追溯、绩效规则可版本化、敏感数据受控和高频查询可索引。本文是数据库实现契约：字段、约束和索引必须由 Flyway 脚本逐项落地，不能在编码时自行增删业务字段。

## 2. 通用约定

1. 业务主键采用雪花算法或数据库生成的 bigint，接口中以字符串传输避免 JavaScript 精度问题。
2. 通用业务表包含主键、创建人、创建时间、更新人、更新时间、逻辑删除标记和版本号。
3. 金额、分数、时长等精确计算数据使用 decimal；时间统一使用 datetime，日期使用 date。
4. 状态、类型等使用受控枚举或字典编码，禁止在业务代码中散落魔法字符串。
5. 外键由应用层和索引保障，生产环境不强制物理外键，以降低跨模块迁移和归档成本。
6. 涉及审批、发布、归档、校准的历史记录不可覆盖更新；采用快照、版本或追加记录保存。

## 4. 关键关系

```text
部门 1---N 员工 N---1 岗位/职级
员工 1---N 人事履历
业务单据 1---1 审批实例 1---N 审批任务/动作记录
员工 1---N 请假余额变化 1---N 请假/加班申请
目标周期 1---N 目标 1---N 目标进度记录
绩效周期 1---N 考核任务 N---1 员工
绩效方案 1---N 方案版本 1---N 方案指标/等级规则；任务引用冻结版本
考核任务 1---N 任务指标快照 1---N 评分记录；任务 1---1 当前结果 1---N 结果版本/校准/申诉记录
```

组织、岗位和员工存在当前有效关系；已办结的人事、假勤和绩效数据保存组织及岗位快照，防止组织调整影响历史报表。

## 5. 关键表设计说明

### 5.1 组织人事

部门表维护树形关系、排序、负责人和有效状态。员工表维护当前任职信息和员工状态，不在员工表中覆盖保存每次异动前后信息。人事异动表与员工履历表记录入职、转正、调动、晋升、停职、离职等事件及流程来源。

员工自然身份信息与任职信息逻辑分层。身份证、银行卡、私人联系方式等敏感信息应加密或脱敏处理；其访问动作写入审计日志。

### 5.2 审批

模板表与节点表定义可配置的线性审批规则。流程实例创建后复制有效节点为实例任务，后续修改模板不影响进行中的流程。动作日志只追加，不更新或删除。业务单据保存流程实例引用与当前状态，但流程任务不直接保存业务规则。

### 5.3 假勤

假期类型定义适用范围、余额规则和审批要求；余额表保存可用余额，余额变动表记录每一次增减的来源。申请通过、撤销、作废等动作必须与余额流水在同一事务内完成。应对同一员工及重叠时间段建立查询索引，作为提交校验支撑。

### 5.4 绩效

绩效方案与指标采用版本化设计。启用考核周期时，将方案项和等级规则快照写入任务或关联快照表。评分记录区分不同评价阶段，最终结果从评分计算而来；校准和申诉用独立记录保存每次变更原因与结果，不能直接覆盖最终分数。

## 6. 索引与查询约定

1. 所有逻辑删除表的高频条件查询应包含 `deleted`，并与组织、状态、时间等条件组成联合索引。
2. 员工目录、待办列表、审批任务、绩效任务、通知列表使用“所属人/组织 + 状态 + 时间”作为主要索引方向。
3. 报表只查询明确时间范围，月度和周期报表优先使用聚合快照表，避免在线扫描全部明细。
4. 模糊搜索限制最小字符数并避免前置通配符；大规模导出走异步任务。
5. 乐观锁字段应用在员工当前信息、余额、目标进度和绩效任务等并发修改对象上。

## 7. 数据生命周期

主数据逻辑删除前必须检查引用关系；被历史业务引用的记录仅可停用。业务附件与实体建立关联后，实体删除不立即删除文件，按保留策略定期清理无引用文件。操作日志、登录日志和归档绩效结果按企业策略长期保存，归档数据可迁移到历史库。

## 8. 数据迁移与初始化

使用 Flyway 管理所有 DDL 与基础字典数据，禁止人工修改生产表结构。初始化数据至少包括管理员账号、基础角色、菜单、组织示例、请假类型、绩效等级和审批模板。每个迁移脚本有明确版本号并在预发布库执行升级、数据校验和恢复演练。生产迁移采用“向前修复 + 备份恢复”策略；不得假定 Flyway 可自动回滚已执行的 DDL。

## 9. 物理模型契约

除只追加日志和报表快照外，所有业务表使用以下字段：`id BIGINT NOT NULL`、`created_by BIGINT`、`created_time DATETIME(3) NOT NULL`、`updated_by BIGINT`、`updated_time DATETIME(3) NOT NULL`、`deleted TINYINT NOT NULL DEFAULT 0`、`version INT NOT NULL DEFAULT 0`。所有时间以 UTC 写入，应用层按企业时区展示。主键为 `BIGINT`，JavaScript 接口一律传字符串。

密码、身份证号、银行卡号使用应用层信封加密后的 `VARBINARY` 或密文 `VARCHAR` 保存；搜索所需的精确匹配值另存不可逆 `*_hash CHAR(64)`，不得对明文建索引。枚举值用 `VARCHAR(32)` 保存，并在本节列出的取值范围内校验。

## 10. 必须由迁移脚本验证的规则

1. 每张表的 `deleted` 查询条件和第一个高频业务条件必须进入联合索引；索引名使用 `uk_`、`idx_` 前缀。
2. 数据库不使用跨模块物理外键，但所有 `*_id` 在应用服务中校验存在性；同模块强一致关系可使用外键，并由迁移脚本统一决定。
3. 关键唯一约束包括工号、用户名、周期-员工考核任务、余额维度、工作日日历、流程业务实例、任务幂等键和结果版本。业务校验不能替代唯一索引。
4. 任何状态迁移必须以 `WHERE id=? AND version=? AND status IN (...)` 更新；影响行数为零时返回并发或状态冲突错误。
