# 08 详细设计

## 1. 文档目的

本文在《02-概要设计》的基础上，定义一期系统各模块的实现边界、关键业务用例、状态机、事务控制、前后端协作和异常处理方式，作为后端与前端编码、联调和测试的共同依据。

本文不重复定义完整表字段、DDL 和 API 请求响应模型。它们分别以《03-数据库设计》和《04-接口设计》为准；页面原型、路由及组件细节以《05-前端设计》为准。

## 2. 模块与代码组织

后端按业务域组织，不以 Controller 或数据库表作为顶层目录。每个业务域包含接口层、应用层、领域层和基础设施实现。

```text
com.hrpm
  common/                 # 统一响应、异常、常量、审计、工具
  security/               # 认证、授权、数据范围、登录会话
  system/                 # 用户、角色、菜单、字典、日志
  organization/           # 部门、岗位、职级、员工档案
  personnel/              # 入职、转正、调动、离职、员工履历
  workflow/               # 模板、实例、审批任务、动作日志
  attendance/             # 请假、加班、余额、汇总
  performance/            # 目标、指标、方案、周期、评分、结果、申诉
  notification/           # 待办、站内信、提醒任务
  report/                 # 报表查询、导出任务、聚合快照
  file/                   # 文件元数据、存储适配、访问授权
```

每个业务域至少包含 `controller`、`application`、`domain`、`repository`、`dto` 和 `assembler`。跨域调用必须通过对方暴露的应用服务或领域事件，禁止直接调用其他模块的 Mapper。

## 3. 横切设计

### 3.1 认证与会话

登录成功后，后端生成短时访问令牌和可续期会话标识。访问令牌携带用户身份和必要的会话版本，权限、数据范围和账号状态从缓存或数据库校验。退出登录、修改密码、禁用账号和角色回收时递增会话版本或删除会话，确保旧令牌立即失效。

请求链路从令牌解析用户身份，填充当前用户上下文。应用服务从上下文取得操作者，不接受客户端传入的操作人作为可信来源。

### 3.2 数据权限

数据范围由角色合并得出，转换为组织范围或本人范围条件后传入 Repository。查询员工、流程、请假、绩效、附件和导出任务均使用统一的数据范围组件。对于通过关联间接访问的数据，先校验主业务对象再读取关联信息。

### 3.3 审计与日志

统一审计切面记录新增、修改、删除、审批、导入、导出、发布、归档、敏感查看和下载行为。日志保存操作者、模块、对象标识、动作、处理结果、来源地址、traceId 和变更摘要。密码、令牌和敏感明文不得写入日志。

### 3.4 幂等、锁与事务

提交类接口使用幂等键或业务状态校验。员工当前任职、假期余额、目标进度和绩效任务使用乐观锁。审批、通过请假、发布绩效、办结离职等动作在单一数据库事务中完成；附件上传、消息通知和报表导出在事务提交后异步执行，避免外部 I/O 导致事务长时间占用。

## 4. 组织与员工详细设计

### 4.1 组织管理

组织维护使用树形结构。新增或移动部门时，服务端校验上级部门有效、不能形成循环、负责人属于有效员工。停用部门前校验是否存在仍在职员工、进行中流程或未迁移的下级组织；不满足条件时拒绝停用并返回业务原因。

部门树读取使用缓存，发生组织调整后主动失效；员工数据范围和审批关系查询不得仅依赖长时间缓存，应使用有效期短的缓存或实时计算。

### 4.2 员工档案

员工档案保存当前有效任职关系。新增档案时校验人员身份唯一性、组织岗位有效性和直属主管关系。员工状态变化不允许由普通编辑接口完成，必须由人事异动用例驱动。查询详情时根据字段权限进行脱敏，并在查看敏感明细时记录审计事件。

### 4.3 人事异动

人事异动按“草稿 - 审批中 - 已通过/已驳回/已撤销 - 已生效”流转。申请提交时冻结员工当前任职快照及审批链。最终通过后，`PersonnelChangeApplicationService` 在一个事务内更新员工当前任职、写入履历、更新异动单状态并创建后续待办。

离职办结需要完成交接确认、资产确认和账号处理检查。离职生效后停止该员工新建业务权限，并按配置保留或回收待处理任务；历史数据仅按权限可读。

## 5. 审批引擎详细设计

### 5.1 核心对象职责

`WorkflowTemplateService` 管理模板和节点配置；`WorkflowStartService` 在业务提交时选择模板并创建实例；`WorkflowTaskService` 处理通过、驳回、退回、撤回等操作；业务模块实现 `WorkflowBusinessHandler` 接口，在流程结束时完成各自生效逻辑。

模板选择顺序为：业务类型匹配、组织范围匹配、启用状态有效、优先级最高。若没有有效模板，业务单据不得提交。启动时保存模板和节点快照，后续模板变更仅影响新实例。

### 5.2 节点推进

1. 发起人提交业务单据，业务模块先完成业务校验。
2. 审批模块创建流程实例和第一个待处理节点，通知审批人。
3. 审批人操作时校验当前任务归属、任务状态、数据范围和单据版本。
4. 通过时关闭当前任务并创建下一任务；最后节点通过时调用业务处理器。
5. 驳回时结束实例并通知发起人；退回修改时将单据恢复为可编辑状态且保留历史意见。
6. 发起人撤回时只允许当前节点未处理且业务规则允许的情况。

业务处理器执行失败时，整个审批动作回滚，当前任务保持待处理状态；不得出现流程显示通过而业务数据未生效的情况。

## 6. 假勤详细设计

### 6.1 请假

请假用例由 `LeaveApplicationService` 编排。保存草稿只校验基础完整性；提交时校验员工状态、申请时间合法性、重叠时间、适用规则、余额与审批模板。审批最终通过时执行余额扣减和余额流水写入，撤销或作废时按原流水反向恢复。

余额调整仅允许 HR 专项权限执行，使用独立动作和审计记录。余额变化使用乐观锁，遇到并发冲突时返回“余额已变化”，要求用户刷新后重试。时间计算由统一工作日服务负责，节假日和最小计量单位由配置提供。

### 6.2 加班与调休

加班申请同样进入审批。通过后若补偿方式为调休，则写入调休余额流水；若为加班费，仅生成供后续薪酬系统读取的确认记录，一期不计算工资。申请撤销、作废和更正必须基于原记录反向处理，不直接修改已确认时长。

### 6.3 月度汇总

月度汇总任务按组织和员工聚合请假、加班和调休数据。任务可重复运行，使用月份与员工维度的幂等键。报表查询优先读取汇总结果，允许在必要时重建指定月份。

## 7. 目标与绩效详细设计

### 7.1 目标管理

目标周期由 HR 创建并发布。部门目标由授权负责人维护，个人目标必须归属于部门目标或按规则说明独立来源。目标进度更新保留更新时间、说明和附件引用；周期冻结后普通用户不能再修改，HR 解冻必须记录理由。

### 7.2 绩效方案与周期

绩效方案由指标、权重、等级规则和评价阶段组成。校验权重总和、指标有效性和等级规则无重叠后才允许启用。启动周期时，系统确定适用员工、复制方案版本并生成唯一考核任务。员工在周期内发生调岗时，默认沿用任务启动时的方案，特殊调整通过 HR 受控操作完成。

### 7.3 考核任务状态机

```text
待自评 -> 待主管评分 -> 待校准 -> 待发布 -> 已发布 -> 已归档
             |                 |
             +----退回自评-----+
已发布 -> 申诉中 -> 已发布（维持或更正）
```

不同阶段仅允许特定角色操作。员工自评提交后锁定自评内容，主管可退回；主管评分提交后进入校准队列；无需校准的方案可直接进入待发布。发布动作批量执行时采用异步任务，每个任务独立处理并汇总失败原因，避免一条异常阻断全周期。

### 7.4 评分与结果

评分计算由 `PerformanceScoreCalculator` 执行：读取任务冻结的指标与权重，计算各项加权结果、总分和等级。所有中间计算结果可追溯，但最终展示按权限控制。HR 校准不直接修改原始评分，而是记录调整决策、理由和调整后的结果。结果发布后，员工可在窗口期确认或申诉；申诉复核若导致更正，则产生新的结果版本并保留原发布版本。

## 8. 通知、待办与文件

待办由业务动作产生，包含处理人、业务类型、对象引用、截止时间和当前状态。通知采用站内信实现，使用模板和变量渲染内容。所有通知发送在业务事务提交后执行；失败任务可重试并记录失败原因。

文件先上传至隔离路径，业务表单提交成功后再建立关联。下载或预览时根据关联业务对象校验权限。未关联临时文件由定时任务清理；已关联附件遵从业务保留策略。

## 9. 报表与导出

看板查询使用受数据范围约束的聚合接口。人员、假勤和绩效报表以周期或月份为查询边界，避免直接扫全表。大数据量导出创建导出任务，后台生成文件后通知申请人；下载链接短时有效，导出记录审计操作人、筛选范围和结果。

## 10. 前后端交互细则

前端为每个业务域封装 API 客户端，使用 TypeScript 类型表示请求和响应。页面只能基于后端返回的状态和操作权限渲染操作区，不自行推断可执行动作。写操作显示加载状态并禁用重复提交；接口返回版本冲突或状态变化时，提示刷新详情。

通用列表封装查询、重置、分页、排序和空状态；通用详情承载状态条、操作区、历史记录和附件区。流程表单可保存草稿，提交后跳转或刷新为只读详情。图表与明细使用同一筛选条件，明细下钻再次调用具备数据权限的后端接口。

## 11. 定时任务与恢复策略

| 任务 | 执行时机 | 幂等与失败处理 |
| --- | --- | --- |
| 试用期/合同提醒 | 每日 | 以员工和提醒类型去重，失败重试并保留日志 |
| 审批与绩效临期提醒 | 每日/按配置 | 根据待办状态生成，已办结任务不再提醒 |
| 月度假勤汇总 | 每月及手工重算 | 以月份和员工为幂等键，允许指定范围重建 |
| 绩效任务初始化 | 周期启动后 | 以周期和员工为唯一约束，重复执行不重复建任务 |
| 通知补偿 | 定时 | 扫描未发送成功通知，指数退避重试 |
| 临时文件清理 | 每日 | 仅清理过期且未关联的文件 |

任务执行记录开始时间、结束时间、结果、处理数量和错误摘要。任务失败告警给管理员；任何任务重跑都必须可审计。

## 12. 安全、异常与验收要点

所有业务接口均执行认证、功能权限和数据权限校验。对敏感读取、导出、下载、权限变更、校准、发布、归档和余额调整记录审计。异常由统一处理器映射为稳定错误码，不向前端暴露堆栈或敏感信息。

详细设计验收重点：模块没有跨层或跨域直接访问；关键状态只经业务动作变更；审批最终通过与业务生效原子一致；并发场景不重复扣减余额、不重复发布结果；前端在权限变化、网络异常、状态变化时给出可恢复的操作反馈。

## 13. 实现不变量与事务边界

以下规则是实现和 Code Review 的强制检查项：

1. 创建或修改业务单据的事务只写本域表、审计日志和 outbox 事件；不得在事务内调用 MinIO、邮件、企业微信或远程 HTTP。事务提交后的任务由 `sys_async_job` 消费。
2. 请假最终通过事务：校验任务和申请状态 -> 乐观锁更新余额 -> 新增不可变余额流水 -> 更新申请状态 -> 写审批动作和 outbox。撤销、作废和更正按原流水生成反向流水，不直接改历史流水。
3. 人事异动最终通过事务：锁定员工当前版本 -> 写履历和前后快照 -> 更新当前任职/状态 -> 更新异动单和流程状态 -> 写待办/通知 outbox。离职还需校验所有 `hr_exit_handover_item.is_required=1` 已确认。
4. 绩效任务创建事务：锁定周期 -> 计算适用员工 -> 插入带 `(cycle_id,employee_id)` 唯一约束的任务 -> 复制方案版本、任务项和组织快照。重复启动返回既有批次或幂等成功，不能生成第二份任务。
5. 绩效发布任务逐人独立事务：确认任务状态与结果版本 -> 创建/更新当前结果指针和新的不可变结果版本 -> 更新任务状态 -> 写通知 outbox。批次只在全部子任务完成或失败已记录后结束。
6. 所有 Repository 写入返回受影响行数；使用乐观锁的更新影响行数不为 1 时必须抛 `VERSION_CONFLICT` 或 `STATE_CONFLICT`，不可重试覆盖。

## 14. 枚举与状态所有权

| 对象 | 状态 | 允许改变状态的应用服务 |
| --- | --- | --- |
| 人事异动 | `DRAFT/IN_PROGRESS/APPROVED/REJECTED/WITHDRAWN/EFFECTIVE` | `PersonnelChangeApplicationService` 与工作流业务处理器 |
| 流程实例/任务 | 实例 `IN_PROGRESS/APPROVED/REJECTED/WITHDRAWN/INVALID`；任务 `PENDING/APPROVED/REJECTED/RETURNED/TRANSFERRED` | `WorkflowTaskService` |
| 请假、加班 | `DRAFT/IN_PROGRESS/APPROVED/REJECTED/CANCELLED/INVALID` | `LeaveApplicationService` / `OvertimeApplicationService` 与流程处理器 |
| 目标周期 | `DRAFT/PUBLISHED/FROZEN/CLOSED` | `GoalCycleApplicationService` |
| 绩效周期 | `DRAFT/ACTIVE/SCORING/CALIBRATING/PUBLISHING/PUBLISHED/CLOSED` | `PerformanceCycleApplicationService` |
| 绩效任务 | `PENDING_SELF/PENDING_MANAGER/PENDING_CALIBRATION/PENDING_PUBLISH/PUBLISHED/APPEALING/ARCHIVED` | `PerformanceTaskApplicationService` |
| 异步作业 | `PENDING/RUNNING/SUCCEEDED/RETRYING/DEAD` | `AsyncJobRunner` |

Controller、Mapper、定时任务和后台管理页面不得直接写上述状态字段。任务和状态动作只能通过表中列出的应用服务执行。
