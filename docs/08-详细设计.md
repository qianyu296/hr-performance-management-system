# 08 详细设计

## 1. 文档目的

本文在《02-概要设计》的基础上，定义当前人力资源管理系统的实现边界、关键业务用例、状态机、事务控制、前后端协作和异常处理方式，作为后端与前端编码、联调和测试的共同依据。

本文不重复定义完整表字段、DDL 和 API 请求响应模型。它们分别以《03-数据库设计》和《04-接口设计》为准；页面原型、路由及组件细节以《05-前端设计》为准。

## 2. 模块与代码组织

后端当前采用按技术职责划分的顶层包，业务实现集中在 Service 并通过 Controller、Mapper、DTO 和 VO 协作。不得跨过 Service 直接从 Controller 调用 Mapper。

```text
com.hrpm
  controller/             # REST 接口与权限注解
  service/                # 部门、员工、人事异动、流程、假勤、系统等业务编排
  mapper/                 # MyBatis 数据访问与数据范围条件
  entity/                 # 领域记录、枚举和持久化模型
  dto/                    # 请求模型
  vo/                     # 响应模型
  security/               # 当前用户、令牌与会话校验
  common/                 # 统一响应、异常、标识生成与链路上下文
```

`PersonnelChangeService` 负责异动单、履历、离职交接和工作流回调的业务边界；`EmployeeService` 负责员工目录、创建、查询和普通档案更新，普通更新不改变任职关系。绩效相关 Controller、Service、Mapper、实体和迁移仍保留在后端，但当前不属于前端产品范围。

## 3. 横切设计

### 3.1 认证与会话

登录成功后，后端生成短时访问令牌和可续期会话标识。访问令牌携带用户身份和必要的会话版本，权限、数据范围和账号状态从缓存或数据库校验。退出登录、修改密码、禁用账号和角色回收时递增会话版本或删除会话，确保旧令牌立即失效。

请求链路从令牌解析用户身份，填充当前用户上下文。应用服务从上下文取得操作者，不接受客户端传入的操作人作为可信来源。

### 3.2 数据权限

数据范围由角色合并得出，转换为组织范围或本人范围条件后传入 Mapper 查询。员工、部门、人事异动、流程和假勤的关联读取先校验主业务对象的可读范围；前端隐藏入口不能替代服务端权限校验。

### 3.3 审计与日志

`OperationAuditService` 当前在异动生效时写入 `PERSONNEL/EFFECTIVE` 操作日志。日志结构可保存操作者、模块、对象标识、动作、处理结果、来源地址、traceId 和变更摘要；其他组织、账号和敏感操作的审计尚未接入该服务，不能作为已交付能力。密码、令牌和敏感明文不得写入日志。

### 3.4 幂等、锁与事务

提交类接口使用业务状态与版本校验。部门、员工、异动单、离职交接事项和假期余额使用乐观锁；影响行数异常时返回冲突而不覆盖。人事异动生效、请假审批和离职办结在单一数据库事务中完成，避免外部 I/O 长时间占用事务。

## 4. 组织与员工详细设计

### 4.1 组织管理

组织维护使用树形结构。新增或移动部门时，服务端校验上级部门有效、不能形成循环、负责人属于有效员工。停用部门前校验是否存在仍在职员工或未迁移的启用下级组织；不满足条件时拒绝停用并返回业务原因。

部门树由 Mapper 实时读取，并按当前用户的组织数据范围裁剪；前端在部门写操作成功或发生版本冲突后重新读取树与当前节点。

### 4.2 员工档案

员工档案保存当前有效任职关系。创建档案时校验工号唯一、组织岗位有效性和直属主管关系；普通更新只允许姓名和性别，且要求版本一致。部门、岗位、职级、直属主管、任职状态、入职与试用日期不允许由普通更新接口改变，必须由人事异动生效用例驱动。`GET /employees/{id}/history` 返回只追加的员工履历，供员工档案抽屉只读展示。

### 4.3 人事异动

人事异动由 `PersonnelChangeService` 处理 `ONBOARD`、`CONFIRM`、`TRANSFER`、`PROMOTION`、`DEMOTION`、`SUSPEND`、`TERMINATION` 七种类型，状态为 `DRAFT`、`IN_PROGRESS`、`APPROVED`、`REJECTED`、`WITHDRAWN`、`EFFECTIVE`。创建草稿时保存异动前后快照；提交时创建 `PERSONNEL_CHANGE` 工作流实例并冻结审批节点。最终审批先将单据转为 `APPROVED`：生效日已到时在同一审批事务中自动生效；未来日期则由具备 `personnel:execute` 的用户调用受版本保护的生效动作，在一个事务内更新当前任职、写入履历并把单据置为 `EFFECTIVE`。

异动列表接口为 `GET /personnel-changes`，支持员工、员工当前部门、类型、状态及生效日期范围筛选。详情响应除了快照、交接事项和版本外，还返回 `canEdit`、`canSubmit`、`canWithdraw`、`canMaintainHandover`、`canExecute`；前端不得自行推断这些操作权限。离职单可通过交接事项接口新增 `WORK`、`ASSET`、`ACCOUNT` 项；事项详情返回 `canConfirm`，确认请求还需要 `personnel:create`，并且仅接收人或具备 `personnel:manage` 的用户可确认。

离职异动生效前，所有必办交接事项必须已确认；生效时更新员工为离职状态、写入离职日期、禁用关联账号并保留按权限可读的历史数据。

## 5. 审批引擎详细设计

### 5.1 核心对象职责

`WorkflowTemplateService` 管理模板和节点配置；各业务 Service（如 `PersonnelChangeService`、`LeaveRequestService`、`OvertimeRequestService`）在提交时选择模板并创建实例；`WorkflowTaskService` 处理通过、驳回、退回、撤回等操作；业务模块实现 `WorkflowBusinessHandler` 接口，在流程结束时完成各自状态更新或生效逻辑。

模板选择顺序为：业务类型匹配、组织范围匹配、启用状态有效、优先级最高。若没有有效模板，业务单据不得提交。启动时保存模板和节点快照，后续模板变更仅影响新实例。

### 5.2 节点推进

1. 发起人提交业务单据，业务模块先完成业务校验。
2. 审批模块创建流程实例和第一个待处理节点，通知审批人。
3. 审批人操作时校验当前任务归属、任务状态和任务版本；最后节点通过时才调用业务处理器执行该业务的额外状态、版本和数据范围校验。
4. 通过时关闭当前任务并创建下一任务；最后节点通过时调用业务处理器。
5. 驳回时结束实例并通知发起人；退回修改时将单据恢复为可编辑状态且保留历史意见。
6. 发起人撤回时只允许当前节点未处理且业务规则允许的情况。

业务处理器执行失败时，整个审批动作回滚，当前任务保持待处理状态；不得出现流程显示通过而业务数据未生效的情况。

## 6. 假勤详细设计

### 6.1 请假

请假用例由 `LeaveRequestService` 和 `LeaveWorkflowService` 编排。保存草稿只校验基础完整性；提交时校验员工状态、申请时间合法性、重叠时间、适用规则、余额与审批模板。审批最终通过时执行余额扣减和余额流水写入，撤销或作废时按原流水反向恢复。

余额调整仅允许 HR 专项权限执行，使用独立动作和审计记录。余额变化使用乐观锁，遇到并发冲突时返回“余额已变化”，要求用户刷新后重试。时间计算由统一工作日服务负责，节假日和最小计量单位由配置提供。

### 6.2 加班与调休

加班申请同样进入审批。通过后若补偿方式为调休，则写入调休余额流水；若为加班费，仅生成供后续薪酬系统读取的确认记录，一期不计算工资。申请撤销、作废和更正必须基于原记录反向处理，不直接修改已确认时长。

### 6.3 月度汇总

月度汇总任务按组织和员工聚合请假、加班和调休数据。任务可重复运行，使用月份与员工维度的幂等键。报表查询优先读取汇总结果，允许在必要时重建指定月份。

## 7. 绩效与目标冻结边界

当前产品定位为人力资源管理系统。目标、绩效指标、方案、周期、评分、校准、发布和申诉的后端 Controller、Service、Mapper、迁移与历史数据保持不变，但前端不注册任何 `/goals/*` 或 `/performance/*` 页面，不在导航、工作台、报表或流程模板中显示入口，也不加载相关前端 API。浏览器直接访问这些路径会由通配路由返回工作台。

恢复绩效 UI 需要独立计划和验收，至少确认人事异动已稳定生效、组织范围授权可验证，以及评分、校准和申诉流程的业务负责人已确认。恢复前不得仅重新暴露单个路由或菜单。

## 8. 通知、待办与文件

待办由业务动作产生，包含处理人、业务类型、对象引用、截止时间和当前状态。通知采用站内信实现，使用模板和变量渲染内容。所有通知发送在业务事务提交后执行；失败任务可重试并记录失败原因。

文件先上传至隔离路径，业务表单提交成功后再建立关联。下载或预览时根据关联业务对象校验权限。未关联临时文件由定时任务清理；已关联附件遵从业务保留策略。

## 9. 报表与导出

数据分析页当前只调用部门人员规模接口，汇总当前启用部门的在职人数。绩效等级分布接口不被前端调用；未来扩展报表或导出时，仍应以数据范围约束查询并记录操作人和筛选范围。

## 10. 前后端交互细则

前端按业务域封装 API 客户端，并以 TypeScript 类型表示请求和响应。人事异动列表使用 `GET /personnel-changes`，详情、创建、更新、提交、撤回、生效、交接新增与交接确认均在 `api/personnel.ts` 中封装；请求和响应中的标识、版本均以字符串传递，后端负责转换与校验。

异动详情只能依据返回的 `can*` 字段和交接事项的 `canConfirm` 渲染操作。写操作显示 `saving` 状态，成功后以完整详情响应替换本地状态；部门维护遇到 `VERSION_CONFLICT` 会重新读取树和当前节点。员工档案将当前任职信息与履历作为只读内容，任职变化跳转到异动草稿而不直接编辑。

## 11. 定时任务与恢复策略

| 任务 | 执行时机 | 幂等与失败处理 |
| --- | --- | --- |
| 试用期/合同提醒 | 每日 | 以员工和提醒类型去重，失败重试并保留日志 |
| 审批临期提醒 | 每日/按配置 | 根据待办状态生成，已办结任务不再提醒 |
| 月度假勤汇总 | 每月及手工重算 | 以月份和员工为幂等键，允许指定范围重建 |
| 通知补偿 | 定时 | 扫描未发送成功通知，指数退避重试 |
| 临时文件清理 | 每日 | 仅清理过期且未关联的文件 |

任务执行记录开始时间、结束时间、结果、处理数量和错误摘要。任务失败告警给管理员；任何任务重跑都必须可审计。

## 12. 安全、异常与验收要点

所有业务接口均执行认证、功能权限和适用的数据权限校验。当前已记录的人事审计动作为异动生效；组织、请假、权限变更和余额调整等审计仍需逐项接入。异常由统一处理器映射为稳定错误码，不向前端暴露堆栈或敏感信息。

详细设计验收重点：Controller 不绕过 Service 访问 Mapper；关键状态只经业务动作变更；异动生效与当前任职、履历、账号状态原子一致；并发场景不重复扣减余额或覆盖版本；前端在权限变化、网络异常、状态变化时给出可恢复的操作反馈；绩效 UI 不得重新出现。

## 13. 实现不变量与事务边界

以下规则是实现和 Code Review 的强制检查项：

1. 创建或修改业务单据的事务只写本域表、审计日志和 outbox 事件；不得在事务内调用 MinIO、邮件、企业微信或远程 HTTP。事务提交后的任务由 `sys_async_job` 消费。
2. 请假最终通过事务：校验任务和申请状态 -> 乐观锁更新余额 -> 新增不可变余额流水 -> 更新申请状态 -> 写审批动作和 outbox。撤销、作废和更正按原流水生成反向流水，不直接改历史流水。
3. 人事异动最终审批先将工作流结束的单据更新为 `APPROVED`；生效日已到时该审批事务继续执行生效，未来生效日则由受控生效动作执行：锁定异动和员工版本 -> 校验离职必办交接已确认 -> 更新当前任职/状态 -> 写入不可变履历 -> 离职时禁用账号并更新异动单为 `EFFECTIVE`。任一环节失败则全部回滚。
4. 所有 Mapper 写入返回受影响行数；使用乐观锁的更新影响行数不为 1 时必须抛 `VERSION_CONFLICT` 或 `STATE_CONFLICT`，不可重试覆盖。

## 14. 枚举与状态所有权

| 对象 | 状态 | 允许改变状态的应用服务 |
| --- | --- | --- |
| 人事异动 | `DRAFT/IN_PROGRESS/APPROVED/REJECTED/WITHDRAWN/EFFECTIVE` | `PersonnelChangeService` 与工作流回调 |
| 流程实例/任务 | 实例 `IN_PROGRESS/APPROVED/REJECTED/WITHDRAWN/INVALID`；任务 `PENDING/APPROVED/REJECTED/RETURNED/TRANSFERRED` | `WorkflowTaskService` |
| 请假、加班 | `DRAFT/IN_PROGRESS/APPROVED/REJECTED/CANCELLED/INVALID` | `LeaveRequestService` / `OvertimeRequestService` 与对应流程处理器 |
| 异步作业 | `PENDING/RUNNING/SUCCEEDED/RETRYING/DEAD` | 持久化模型已定义；当前未提供独立作业运行器实现 |

Controller、Mapper、定时任务和后台管理页面不得直接写上述状态字段。任务和状态动作只能通过表中列出的应用服务执行。
