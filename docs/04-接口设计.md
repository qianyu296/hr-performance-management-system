# 04 接口设计

## 1. 设计原则

API 采用 REST 风格，统一前缀为 `/api/v1`，JSON 使用 UTF-8。接口按资源和业务域划分，读操作使用 GET，创建使用 POST，受控修改使用 PUT/PATCH，逻辑删除使用 DELETE。审批、发布、归档等状态动作使用明确的动作型子资源，避免通用更新绕过业务规则。

## 2. 通用约定

1. 登录后的请求携带 `Authorization: Bearer <token>`；文件上传按接口要求使用 multipart/form-data。
2. 成功响应统一包含业务状态、提示、数据和 traceId；失败响应包含稳定错误码与可读信息。
3. 列表响应统一包含 records、total、page 和 pageSize；默认分页并限制最大 pageSize。
4. 写操作支持 `Idempotency-Key`，尤其是提交申请、发布结果、导入和异步导出。
5. 请求校验失败返回参数级错误；权限不足、数据范围不足、状态冲突和乐观锁冲突使用不同错误码。
6. 服务端过滤所有未授权字段；前端不可通过传参突破数据范围。

## 3. API 分组

| 分组 | 路径示例 | 主要能力 |
| --- | --- | --- |
| 认证与个人中心 | /auth、/me | 登录、登出、刷新会话、当前用户和待办摘要 |
| 系统管理 | /users、/roles、/menus、/dictionaries | 用户、角色、菜单、字典与审计查询 |
| 组织人事 | /departments、/positions、/employees、/personnel-changes | 组织、任职、档案、履历和异动流程 |
| 假勤 | /leave-types、/leave-balances、/leave-requests、/overtime-requests | 规则、余额、申请、撤销与统计 |
| 目标绩效 | /performance/cycles、/goals、/schemes、/tasks、/results、/appeals | 周期、目标、方案、考核任务、结果与申诉 |
| 审批协同 | /workflow/templates、/workflow/tasks、/notices、/files | 模板、待审批任务、通知和附件 |
| 报表导出 | /reports、/exports | 看板数据、明细查询和异步导出 |

## 4. 关键动作接口

| 业务 | 动作 | 说明 |
| --- | --- | --- |
| 人事异动 | 提交、撤回、审批、办结 | 由流程状态决定是否允许执行 |
| 请假/加班 | 保存草稿、提交、撤销、审批 | 提交与通过时由服务端执行规则校验 |
| 目标 | 发布、更新进度、冻结、解冻 | 解冻必须具备授权和理由 |
| 绩效 | 启动周期、自评提交、主管评分、校准、发布、确认、申诉、归档 | 每个动作只允许特定任务状态和角色执行 |
| 文件 | 上传、下载、预览、删除关联 | 下载和预览须校验关联业务的数据权限 |

动作请求必须传递对象版本或等价并发控制信息。服务端返回当前状态与下一步可执行动作，便于前端刷新显示。

## 5. 错误码分层

| 类别 | 示例 | 前端处理 |
| --- | --- | --- |
| 认证 | 未登录、令牌失效、账号禁用 | 跳转登录或刷新会话 |
| 授权 | 无菜单权限、无数据权限、无审批权 | 提示无权并返回安全页面 |
| 校验 | 必填缺失、格式错误、时间冲突 | 标记输入位置并保留表单 |
| 状态 | 已撤销、已归档、审批节点变化 | 刷新详情并提示最新状态 |
| 并发 | 数据版本过期、余额已变动 | 提示刷新后重新操作 |
| 系统 | 文件服务异常、导出失败、未知异常 | 显示 traceId，支持重试或联系管理员 |

## 6. 安全与审计

除登录、健康检查和明确公开资源外，所有接口都需要身份认证。敏感查询、导出、下载、审批、发布和权限变更写入审计日志。对高频列表、报表和导出接口实施限流；导出任务必须异步化并设置下载时效。

## 7. 接口测试要求

每个业务域至少覆盖：认证失败、权限不足、正常路径、参数校验、状态非法、并发冲突和数据范围隔离。接口契约使用 OpenAPI 文档生成，并由前端在联调前确认响应模型。

## 8. 可执行契约

接口定义文件固定为 `docs/hrpm-v1.openapi.yaml`，以 OpenAPI 3.1 编写并纳入版本控制。本文定义资源、动作和字段口径；实现前必须从该文件生成后端接口桩和前端 TypeScript 类型。本文与 OpenAPI 不一致时，以通过契约测试的 OpenAPI 为准，并同步修改本文。

### 8.1 统一请求和响应

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "data": {},
  "traceId": "01J..."
}
```

所有 `id`、`version`、分页游标和时间使用字符串传输。写入 DTO 不允许包含 `createdBy`、`updatedBy`、操作人、数据范围或任何服务端派生字段。除登录、刷新令牌和健康检查外，请求必须携带 Bearer 令牌；变更操作必须携带 `Idempotency-Key`，且 Body 中携带当前对象 `version`。

列表使用 `page`（从 1 开始）、`pageSize`（1-100）、`sortBy` 和 `sortOrder`；响应为 `{records,total,page,pageSize}`。时间字段采用 ISO-8601 UTC，日期字段为 `yyyy-MM-dd`。附件只传 `fileId`，不得向业务接口传对象存储地址。

### 8.3 动作命令和状态前置条件

`approve/reject/return/transfer` 统一请求为 `{version, comment, transferToUserId?}`。`reject` 结束流程，`return` 将业务单据变为可编辑状态，`transfer` 必须记录原处理人和原因。审批任务不能由业务单据接口直接更新。

`submit` 请求只能从 `DRAFT` 或 `RETURNED` 执行；`cancel/withdraw` 只能从 `IN_PROGRESS` 执行且当前节点未处理；`complete` 只能由工作流最后节点的业务回调调用。所有动作响应使用：`{id, version, status, workflowInstanceId, availableActions, updatedTime}`。

绩效评分请求为 `{version, items:[{taskItemId, score, comment, fileIds}]}`；服务端重新计算加权分和总分，客户端不得传最终等级。校准请求必须包含 `{version, adjustedScore?, adjustedLevel?, reason}`，至少有一个调整值且 `reason` 非空。申诉处理若改变结果，必须返回新 `resultVersion`。

### 8.4 稳定错误码

| 错误码 | HTTP | 含义 |
| --- | --- | --- |
| `AUTH_UNAUTHENTICATED` / `AUTH_SESSION_INVALID` | 401 | 未登录、令牌失效或会话版本失效 |
| `AUTH_FORBIDDEN` / `AUTH_DATA_SCOPE_DENIED` | 403 | 功能、字段或数据范围不足 |
| `VALIDATION_FAILED` / `TIME_CONFLICT` / `BALANCE_INSUFFICIENT` | 400 | 参数、时间冲突或余额规则失败 |
| `RESOURCE_NOT_FOUND` | 404 | 在当前授权范围内不存在资源 |
| `IDEMPOTENCY_CONFLICT` / `VERSION_CONFLICT` / `STATE_CONFLICT` | 409 | 重复提交、乐观锁或状态已变化 |
| `WORKFLOW_TEMPLATE_MISSING` / `WORKFLOW_TASK_INVALID` | 422 | 模板缺失或审批任务不可处理 |
| `ASYNC_JOB_FAILED` / `FILE_QUARANTINED` | 422 | 异步任务失败或文件不可用 |
| `SYSTEM_ERROR` | 500 | 未预期异常，响应只暴露 traceId |

### 8.5 核心写入字段契约

标记 `R` 的字段为提交必填；`C` 表示由特定状态或类型决定是否必填；未列字段一律由服务端生成或拒绝。各资源的普通 PATCH 均另含 `version`，动作型请求还需 `Idempotency-Key`。

| 命令 | 必填字段 | 条件字段与服务端规则 |
| --- | --- | --- |
| 用户创建 | `username(R)`、`employeeId(C)`、`roleIds(R)`、`status(R)` | 初始密码由服务端生成/重置；离职员工不能启用账号 |
| 部门/岗位/职级 | `code(R)`、`name(R)`、`status(R)` | 部门另需 `parentId(C)`、`leaderEmployeeId(C)`；停用前服务端验证引用 |
| 员工档案 | `employeeNo(R)`、`name(R)`、`departmentId(R)`、`positionId(R)`、`employmentStatus(R)`、`hireDate(R)` | `managerEmployeeId(C)`；证件、电话为受控字段；不得以普通更新改变任职状态 |
| 人事异动 | `employeeId(R)`、`changeType(R)`、`effectiveDate(R)`、`reason(R)` | 调动/晋降需 `afterDepartmentId/afterPositionId/afterRankId(C)`；离职需 `lastWorkingDate(R)`、`handoverId(R)` |
| 请假申请 | `leaveTypeId(R)`、`startTime(R)`、`endTime(R)`、`reason(R)` | `duration` 由服务端计算；满足请假类型规则时 `fileIds(R)`；提交时校验日历、重叠、余额和模板 |
| 加班申请 | `startTime(R)`、`endTime(R)`、`reason(R)`、`compensationType(R)` | `compensationType` 仅 `TIME_OFF/OVERTIME_PAY`；时长由服务端以起止时间计算；`TIME_OFF` 审批通过写调休余额流水，撤销写反向流水，`OVERTIME_PAY` 仅确认加班记录供薪酬系统读取 |
| 余额调整 | `deltaHours(R)`、`direction(R)`、`reason(R)`、`version(R)` | `direction` 仅 `INCREASE/DECREASE`；仅 `attendance:balance:adjust` 专项授权；必须创建流水，禁止直接 PATCH 余额 |
| 目标 | `goalCycleId(R)`、`name(R)`、`ownerEmployeeId(R)`、`targetValue(R)`、`unit(R)`、`weight(R)`、`dueDate(R)` | `parentGoalId(C)`；权重在同级口径内校验；进度使用独立 `{progress(R),description(R),fileIds(C)}` 命令 |
| 方案版本 | `schemeId(R)`、`versionName(R)`、`evaluationStages(R)`、`items(R)`、`levelRules(R)` | 每个 item 含 `metricId(R),weight(R),scoreMethod(R)`；服务端校验权重 100、等级不重叠 |
| 绩效周期启动 | `schemeVersionId(R)`、`startDate(R)`、`endDate(R)`、`selfDeadline(R)`、`managerDeadline(R)`、`applicabilityRule(R)` | 服务端解析适用员工并创建异步批次；不得由客户端传 employeeIds 绕过范围，个别补发走 HR 专项动作 |
| 自评/主管评分 | `items(R)` | 每项含 `taskItemId(R),score(C),comment(C),fileIds(C)`；是否要求分数/评语由任务项快照决定 |
| 校准 | `reason(R)` | `adjustedScore(C)` 或 `adjustedLevel(C)` 至少一个；仅 HR 负责人或专项授权角色可执行 |
| 申诉与结论 | 申诉：`appealItem(R),statement(R)`；结论：`decision(R),conclusion(R)` | 申诉附件条件必填；`decision=ADJUST` 必须生成新结果版本 |
