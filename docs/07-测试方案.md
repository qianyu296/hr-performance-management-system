# 07 测试方案

## 1. 测试目标

验证系统在功能正确性、权限隔离、流程一致性、数据安全、稳定性和可用性方面满足需求。测试覆盖后端单元与集成测试、接口测试、前端组件测试、端到端测试、性能测试和安全测试。

## 2. 测试环境与数据

测试环境应与生产环境的中间件版本保持一致，使用独立数据库、Redis 和对象存储桶。测试数据必须使用虚构员工和脱敏附件，不允许复制真实人事数据。准备多个部门层级、多个角色、在职与离职员工、不同绩效状态和不同假期余额，覆盖权限、组织变化和异常路径。

## 3. 测试分层

| 层级 | 工具建议 | 覆盖重点 |
| --- | --- | --- |
| 单元测试 | JUnit 5、Mockito | 状态机、绩效计算、余额计算、权限规则、日期规则 |
| 集成测试 | Spring Boot Test、Testcontainers | 事务、Mapper、Redis、文件适配和应用服务编排 |
| API 测试 | MockMvc、Postman/Newman | 鉴权、参数校验、数据范围、错误码、幂等和并发 |
| 前端测试 | Vitest、Vue Test Utils | 组件状态、权限指令、表单校验、请求拦截 |
| E2E 测试 | Playwright | 登录、流程提交审批、绩效、权限隔离和导出体验 |
| 性能与安全 | JMeter/k6、依赖扫描 | 并发查询、导出任务、鉴权、越权、敏感数据保护 |

## 4. 核心测试场景

### 4.1 认证与权限

1. 未登录、令牌过期、禁用账号和错误密码均得到正确处理。
2. 普通员工无法读取他人档案、申请、绩效和附件；主管只能看到授权团队。
3. 删除或收回角色后，用户刷新会话即失去对应菜单、按钮和接口能力。
4. 敏感字段默认脱敏；无专项权限的导出不包含敏感内容。

### 4.2 组织与人事

1. 组织树不能形成循环；停用组织不能再被新业务引用。
2. 入职、转正、调动、离职按照审批结果改变当前状态并形成可查询履历。
3. 员工调岗或离职后，历史流程和绩效结果仍显示正确快照。
4. 批量导入遇到重复或引用错误时，错误项被反馈且合法数据按明确规则处理。

### 4.3 假勤

1. 重叠时间、余额不足、无审批模板、已离职员工提交等情况必须被拒绝。
2. 申请审批通过、撤销和作废后，余额及变动流水保持一致。
3. 多人同时操作余额或同一单据时，系统不产生重复扣减或重复审批。
4. 月度汇总与申请明细口径一致。

### 4.4 目标绩效

1. 启动周期后为适用员工生成唯一任务，方案版本被冻结。
2. 员工自评、主管评分、HR 校准、发布和归档只能按状态顺序进行。
3. 权重计算、等级映射、边界分数和舍入结果符合规则。
4. 发布前员工不可见最终结果；发布后仅可查看授权范围。
5. 申诉、复核和归档更正均留下完整审计记录，不能覆盖原始事实。

## 5. 非功能测试

1. 对常用列表、详情、审批和登录进行并发压测，验证常规响应时间目标。
2. 对月度汇总和导出验证异步执行、进度、失败重试及下载权限。
3. 执行 SQL 注入、越权访问、ID 枚举、文件类型伪造、令牌重放和 XSS 测试。
4. 验证异常日志不泄漏密码、令牌、身份证号等敏感信息。
5. 验证数据库备份恢复、附件可访问性、定时任务幂等和服务重启后的恢复行为。

## 6. 缺陷管理与发布准入

缺陷按阻断、严重、一般和建议分级。阻断和严重缺陷必须关闭后才能发布；一般缺陷需有明确规避方案和计划。发布前必须完成数据库迁移演练、回归测试、接口契约检查、前端构建检查、关键 E2E 场景和权限安全检查。

## 7. 验收清单

1. 核心业务链路：员工档案、人事异动、请假审批、绩效自评到发布、申诉处理均可闭环。
2. 角色和数据范围满足权限设计，接口与页面均无越权。
3. 状态流转、余额与绩效结果在异常或并发下保持一致。
4. 关键操作、敏感查看和导出具备审计记录。
5. 前端在常用桌面浏览器下无阻塞性布局和交互问题，接口失败有明确反馈。
6. 自动化测试、构建、数据库迁移和文档检查均通过后方可交付。

## 8. 可执行质量门禁

测试用例使用 `TC-<域>-<序号>` 编号，并在用例中关联一个或多个 `REQ-<域>-<序号>` 与 OpenAPI `operationId`。任何 R1/R2 需求没有至少一个正向、一个越权或非法状态用例时，不得进入验收。缺陷单必须关联测试用例、接口 traceId 和数据库迁移版本。

| 质量项 | R1 准入标准 | R2 增补标准 |
| --- | --- | --- |
| 单元测试 | 状态机、余额、评分、权限规则覆盖率不低于 80% 行覆盖和 70% 分支覆盖 | 对版本更正、目标冻结、模板选择和异步重试维持同一标准 |
| 接口契约 | OpenAPI 校验通过；所有写操作包含幂等、版本冲突和授权测试 | 生成的前端类型与后端契约兼容性检查通过 |
| E2E | 登录、请假提交审批、绩效自评评分发布、数据范围隔离通过 | 离职交接、加班调休、目标树、校准申诉、导出恢复通过 |
| 性能 | 100 并发用户下，列表 P95 <= 2 秒、写操作 P95 <= 3 秒，错误率 < 1% | 500 并发读用户下同一指标；10 万条历史明细的月报不直接扫描主表 |
| 异步 | 重复执行不产生重复任务或流水；失败后可重试 | 进程在 outbox 写入后、消费前和消费途中终止时均可恢复 |
| 安全 | 未授权、ID 枚举、SQL 注入、XSS、令牌失效、文件越权均为零高危 | 敏感字段导出、权限回收即时生效、恶意文件隔离与审计通过 |
| 恢复 | 预发布完成 MySQL 与附件恢复演练，恢复结果可查询 | 在目标 RPO/RTO 内完成恢复演练并保留报告 |

### 8.1 最小测试数据集

准备根部门、两层子部门、跨部门调动员工、直属主管链、HR、财务和高层账号；至少包含在职、试用、离职员工，余额充足/不足/并发竞争三种假期余额，工作日/周末/节假日，启用与停用的模板，进行中与归档的绩效周期，以及已发布后申诉更正的结果版本。所有数据由 Flyway 的测试种子或 Testcontainers fixture 创建，测试结束可重复清理。
