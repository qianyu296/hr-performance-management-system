# 07 测试方案

## 1. 测试目标

验证当前人力资源管理系统在功能正确性、权限隔离、流程一致性和可用性方面满足交付要求。自动化覆盖后端 JUnit 5、Spring Boot Test、MockMvc、Flyway 迁移与前端 TypeScript/Vite 构建；关键角色和人事流程补充人工验收。

## 2. 测试环境与数据

测试环境使用独立 MySQL 数据库；Testcontainers 迁移测试在具备 Docker 时执行。测试数据必须使用虚构员工，准备部门层级、HR/部门主管/普通员工角色、在职与离职员工、请假余额及人事异动快照。目标与绩效仅保留后端历史兼容性，不属于当前 UI 验收范围。

## 3. 测试分层

| 层级 | 工具建议 | 覆盖重点 |
| --- | --- | --- |
| 单元测试 | JUnit 5、Mockito | 状态机、余额计算、权限规则、日期规则 |
| 集成测试 | Spring Boot Test、Testcontainers | Flyway、Mapper、事务、服务编排与人员数据范围 |
| API 测试 | MockMvc、Postman/Newman | 鉴权、参数校验、数据范围、错误码、幂等和并发 |
| 前端验证 | `vue-tsc`、Vite | 页面类型、模板类型和生产包构建 |
| 人工验收 | 浏览器与三类测试账号 | 登录、组织范围、人事异动、交接、权限菜单与路由冻结 |

## 4. 核心测试场景

### 4.1 认证与权限

1. 未登录、令牌过期、禁用账号和错误密码均得到正确处理。
2. 普通员工无法读取他人档案、异动和申请；主管只能看到授权团队。
3. 删除或收回角色后，用户刷新会话即失去对应菜单、按钮和接口能力。
4. 敏感字段默认脱敏；无专项权限的导出不包含敏感内容。

### 4.2 组织与人事

1. 组织树不能形成循环；停用组织不能再被新业务引用。
2. 入职、转正、调动、晋升、降职、停职和离职按照审批与生效日期改变当前状态，并形成不可修改的履历。
3. 员工调岗或离职后，异动快照、流程历史和审计记录保持可查询。
4. 离职必办交接事项未确认时，生效被拒绝；离职生效后账号禁用且旧会话失效。
5. 部门循环移动、无效负责人、仍有在职员工的停用和乐观锁冲突均返回明确错误。

### 4.3 假勤

1. 重叠时间、余额不足、无审批模板、已离职员工提交等情况必须被拒绝。
2. 申请审批通过、撤销和作废后，余额及变动流水保持一致。
3. 多人同时操作余额或同一单据时，系统不产生重复扣减或重复审批。
4. 月度汇总与申请明细口径一致。

### 4.4 前端范围冻结

1. 导航、工作台、报表和流程模板页面不显示目标、绩效、自评、评分或绩效申诉入口。
2. 直接访问 `/goals/*`、`/performance/*` 命中通配路由并返回工作台，不加载相关 Vue 页面或 API。
3. `PERFORMANCE_APPEAL` 历史模板记录保留在数据库，但不出现在模板列表、新建业务类型和前端筛选中。

## 5. 非功能测试

1. 对常用列表、详情、审批和登录进行并发压测，验证常规响应时间目标。
2. 对月度汇总和导出验证异步执行、进度、失败重试及下载权限。
3. 执行 SQL 注入、越权访问、ID 枚举、文件类型伪造、令牌重放和 XSS 测试。
4. 验证异常日志不泄漏密码、令牌、身份证号等敏感信息。
5. 验证数据库备份恢复、附件可访问性、定时任务幂等和服务重启后的恢复行为。

## 6. 缺陷管理与发布准入

缺陷按阻断、严重、一般和建议分级。阻断和严重缺陷必须关闭后才能发布；一般缺陷需有明确规避方案和计划。发布前必须完成数据库迁移演练、回归测试、接口契约检查、前端构建检查、关键 E2E 场景和权限安全检查。

## 7. 验收清单

1. 核心业务链路：组织维护、员工档案、人事异动、请假和加班审批均可闭环。
2. 角色和数据范围满足权限设计，接口与页面均无越权。
3. 人事状态流转、交接、余额在异常或并发下保持一致。
4. 已接入的人事异动生效操作具备审计记录；敏感查看、导出等其他审计事件需在对应功能接入后单独验收。
5. 前端在常用桌面浏览器下无阻塞性布局和交互问题，接口失败有明确反馈。
6. 自动化测试、构建、数据库迁移和文档检查均通过后方可交付。

## 8. 可执行质量门禁

测试用例使用 `TC-<域>-<序号>` 编号，并在用例中关联一个或多个 `REQ-<域>-<序号>` 与 OpenAPI `operationId`。任何 R1/R2 需求没有至少一个正向、一个越权或非法状态用例时，不得进入验收。缺陷单必须关联测试用例、接口 traceId 和数据库迁移版本。

| 质量项 | R1 准入标准 | R2 增补标准 |
| --- | --- | --- |
| 自动化回归 | `mvn test` 通过；人事异动、部门、员工、请假、加班、权限和迁移测试覆盖关键路径 | Docker 可用时 Testcontainers 迁移测试执行；当前 Docker 不可用时记录跳过原因 |
| 前端 | `npm.cmd run typecheck` 与 `npm.cmd run build` 通过 | 引入前端测试框架后补充组件与 E2E 测试 |
| 人工验收 | HR、部门主管、普通员工完成目录范围、异动、交接、账号失效、履历、审批历史和越权拒绝验证 | 绩效 UI 恢复前单独建立验收计划 |
| 性能 | 100 并发用户下，列表 P95 <= 2 秒、写操作 P95 <= 3 秒，错误率 < 1% | 500 并发读用户下同一指标；10 万条历史明细的月报不直接扫描主表 |
| 异步 | 重复执行不产生重复任务或流水；失败后可重试 | 进程在 outbox 写入后、消费前和消费途中终止时均可恢复 |
| 安全 | 未授权、ID 枚举、SQL 注入、XSS、令牌失效、文件越权均为零高危 | 敏感字段导出、权限回收即时生效、恶意文件隔离与审计通过 |
| 恢复 | 预发布完成 MySQL 与附件恢复演练，恢复结果可查询 | 在目标 RPO/RTO 内完成恢复演练并保留报告 |

### 8.1 最小测试数据集

准备根部门、两层子部门、跨部门调动员工、直属主管链、HR 和普通员工账号；至少包含在职、试用、停职和离职员工，余额充足/不足/并发竞争三种假期余额，工作日/周末/节假日，以及启用与停用的流程模板。所有数据由测试夹具或 Flyway 种子创建，测试结束可重复清理。
