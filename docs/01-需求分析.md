# 基于 Spring Boot 的人力资源与绩效管理系统

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | 人力资源与绩效管理系统（HRPM） |
| 系统定位 | 面向中小企业的人力资源数字化与绩效考核管理平台 |
| 使用对象 | 企业管理层、人力资源部、部门负责人、普通员工、财务人员（只读） |
| 建设目标 | 统一员工主数据和组织架构，在线完成常用人事流程与绩效考核，提供可追溯的管理数据和统计分析。 |
| 产品范围 | 组织员工、入转调离、请假与加班、绩效方案与考核、目标管理、通知待办、报表、权限与审计。 |
| 发布基线 | R1（MVP）与 R2（完整一期）分两次发布；每项能力必须归属一个发布基线。 |
| 版本 | V1.0 |

## 2. 建设背景与痛点

中小企业常用 Excel、即时通信工具和纸质单据管理员工信息、审批及绩效结果，容易出现下列问题：

1. 员工档案分散，部门、岗位、汇报关系和在职状态更新不同步。
2. 入职、转正、调岗和离职审批缺少统一记录，交接事项容易遗漏。
3. 请假、加班依赖聊天记录或纸质单据，审批状态、余额和统计口径不一致。
4. 绩效指标、评分过程与最终结果不透明，缺少考核依据、申诉记录和历史归档。
5. 管理者无法及时掌握人员结构、流动情况、考勤异常及绩效分布。
6. 敏感人事数据权限过宽，缺少关键操作的审计追踪。

系统应以“员工全生命周期”和“目标到绩效结果”的业务闭环为核心，减少重复录入，使员工、主管、HR 在各自权限范围内协同完成业务。

## 3. 建设目标

### 3.1 业务目标

1. 建立唯一、可维护的组织和员工主数据。
2. 线上闭环管理员工入职、转正、调动、离职等人事事件。
3. 使请假、加班等申请具有明确状态、审批记录和可统计数据。
4. 支持按周期、部门和岗位发布绩效方案，完成自评、主管评分、校准、发布和确认。
5. 将部门目标逐级分解到员工，并关联绩效评价依据。
6. 向管理层提供实时、统一口径的人员与绩效分析。

### 3.2 系统目标

1. 基于角色和组织范围控制菜单、功能和数据访问。
2. 所有关键流程节点保留申请内容、审批意见、操作人和操作时间。
3. 支持附件上传、站内通知、待办提醒、Excel 导入导出。
4. 保证历史业务数据在组织调整、员工离职和方案变更后仍可追溯。
5. 为后续接入考勤机、企业微信/钉钉、单点登录和薪酬系统预留扩展能力。

### 4.2 不纳入一期的范围

1. 物理考勤机、门禁、GPS 打卡等设备直连；一期以人工导入或申请数据汇总为主。
2. 复杂薪资核算、个税申报、社保公积金申报和银行代发。
3. 招聘门户、候选人简历解析、线上面试。
4. 完整培训学习平台和在线考试。
5. 多法人、多币种、多国家劳动法规核算。

这些能力可在系统稳定后以独立模块或第三方接口方式接入。

## 5. 用户角色与权限边界

| 角色 | 主要职责 | 数据范围 |
| --- | --- | --- |
| 系统管理员 | 维护账号、角色、菜单、字典和系统参数；查看日志 | 全部系统数据，不直接处理业务审批 |
| HR 专员 | 维护组织、岗位、员工档案，发起和处理人事流程，配置考核周期 | 全公司人事与绩效数据；敏感字段按授权可见 |
| HR 负责人 | 审批重要人事变动，校准绩效结果，发布人事及绩效报表 | 全公司数据 |
| 部门负责人 | 维护本部门目标，审批下属申请，对下属进行绩效评分 | 本部门及下级部门 |
| 直属主管 | 审批直属员工申请，设定/确认员工目标，完成评分与反馈 | 直属及授权团队 |
| 普通员工 | 维护本人有限资料，提交申请，填写自评，查看本人结果与待办 | 仅本人数据 |
| 财务人员 | 查询已发布的人员、岗位、绩效等级和薪酬测算基础数据 | 经授权的只读字段和范围 |
| 高层管理者 | 查看经营层面的人员与绩效分析 | 全公司聚合数据；默认不展示个人敏感信息 |

### 5.1 权限模型

系统采用 RBAC（角色权限控制）结合数据权限：

1. 菜单权限：控制用户可以看到哪些模块。
2. 按钮权限：控制新增、编辑、删除、导入、导出、审批、发布等操作。
3. 数据权限：支持“本人、直属下属、本部门、本部门及下级部门、全公司、自定义部门”范围。
4. 字段权限：身份证号、银行卡、联系方式、薪酬测算等敏感字段需要脱敏显示或单独授权。
5. 流程权限：审批人由组织汇报关系、岗位、指定人员或 HR 规则决定，并在提交时固化审批链。

## 6. 核心业务流程

### 6.1 员工生命周期

1. HR 创建候选入职记录，填写基本资料、所属部门、岗位、职级、直属主管、合同信息和入职日期。
2. 入职申请经部门负责人和 HR 审批后，系统创建员工档案与登录账号；账号初始状态可设为待激活。
3. 试用期到期前，系统向员工、直属主管和 HR 发送转正提醒。员工或主管发起转正申请，填写试用期总结与评价。
4. 调动申请可变更部门、岗位、职级、直属主管、生效日期和原因；审批完成后写入员工履历，原归属保留为历史记录。
5. 离职申请需填写离职原因、最后工作日、工作交接、资产归还和账号停用计划；审批完成后员工状态变更为离职，账号在指定日期禁用。

员工状态：`待入职`、`试用`、`正式`、`停职`、`离职`。状态变更只能由已完成的业务流程触发，禁止直接覆盖历史。

### 6.2 请假与加班流程

1. HR 配置请假类型、适用人员、是否扣减余额、年度额度和最小申请单位。
2. 员工提交请假申请，填写起止时间、时长、原因和证明附件；系统校验时间冲突、余额与工作日规则。
3. 员工提交加班申请，填写日期、时段、预计时长、事由和补偿方式（调休或加班费）。
4. 系统根据申请人组织关系生成审批链；审批人可通过、驳回或退回修改。
5. 已通过申请写入假勤明细并影响余额/调休余额；申请人在未开始或规则允许的情况下可以撤销。
6. HR 每月生成部门和个人汇总，识别未审批、超额请假、连续加班等异常。

申请状态：`草稿`、`审批中`、`已通过`、`已驳回`、`已撤销`、`已失效`。

### 6.3 目标管理流程

1. HR 创建目标周期，例如“2026 年第三季度”。
2. 高层或部门负责人创建部门目标，设置目标名称、衡量口径、目标值、权重、负责人和截止日期。
3. 部门负责人将目标拆解为个人目标；一个个人目标可关联一个上级目标，形成目标树。
4. 员工定期更新进度、产出说明和附件；直属主管确认或反馈。
5. 周期结束后冻结目标数据，作为绩效评分的重要依据；仅 HR 可在受控条件下解冻并记录原因。

### 6.4 绩效考核流程

1. HR 创建考核周期，定义周期范围、适用部门/岗位、评分时间、结果发布时间和申诉截止时间。
2. HR 或部门负责人选择绩效方案。方案由指标项、权重、评分规则、等级规则和评价人构成。
3. 系统为适用员工生成考核任务。员工在截止日前完成自评，填写每项完成情况、证据附件和总结。
4. 直属主管依据目标完成情况和指标说明评分，可填写优势、改进项和发展建议。
5. HR 负责人对结果进行校准，可调整等级或分数；调整必须填写原因并保留调整前后数据。
6. 系统按权重计算总分并映射等级，例如 A、B、C、D。结果经批准后统一发布。
7. 员工确认结果；若有异议，可在截止前提交申诉。HR 分派复核人并记录处理结论。
8. 周期关闭后归档，结果只读；后续周期不能修改历史结果。

绩效任务状态：`待自评`、`待主管评分`、`待校准`、`待发布`、`已发布`、`申诉中`、`已归档`。

## 7. 功能需求

### 7.1 基础配置与组织管理

1. 公司组织支持树形结构，可维护部门编码、名称、负责人、上级部门、启停状态和生效日期。
2. 岗位和职级支持编码、名称、所属序列、岗位说明、等级排序和启停状态。
3. 员工档案至少包含工号、姓名、性别、证件信息、联系方式、入职日期、部门、岗位、职级、主管、员工状态、合同到期日等。
4. 工号必须唯一；离职员工工号不可复用。部门、岗位、主管变更需要形成履历记录。
5. 支持按部门、岗位、职级、状态、入职日期、姓名和工号组合查询。
6. 支持批量导入员工；导入前校验必填字段、编码引用、重复工号和数据格式，并输出错误明细。

### 7.2 人事异动管理

1. 支持入职、转正、调动、晋升、降职、停职、离职等异动类型。
2. 各异动记录须保存申请人、申请时间、生效日期、异动前后信息、原因、附件、审批流程与结果。
3. 转正申请应展示试用期起止时间、员工总结、主管评价、转正建议和最终决定。
4. 离职流程应包含工作交接人、交接事项、资产归还确认、账号关闭确认；未完成关键项时不允许办结。
5. 支持查询员工完整履历，并按异动类型、日期和状态筛选。

### 7.3 请假与加班管理

1. 请假规则支持年假、病假、事假、婚假、产假等配置；一期的法定规则采用参数化配置，不内置地区法规自动计算。
2. 系统以半天或小时为可配置最小单位，自动计算申请时长，并排除休息日/节假日配置。
3. 提交时须校验同一员工的时间重叠、剩余余额、提交截止规则和必填附件。
4. 支持管理个人假期余额、调休余额及手工调整；手工调整必须记录原因和操作人。
5. 支持按人员、部门、类型、月份、状态统计请假与加班时长。

### 7.4 目标与绩效方案管理

1. 指标库支持 KPI、OKR、行为评价等指标类型，包含指标名称、说明、单位、评分方式、评分区间、默认权重和适用岗位。
2. 绩效方案可按部门、岗位或人员配置；一个周期内员工只能命中一个已启用方案。
3. 权重之和必须为 100%；评分项支持数值评分、等级评分和主管评语。
4. 等级规则支持按分数区间映射，例如 A：90-100，B：80-89；规则在周期启用后需要版本冻结。
5. 支持设置评价人：员工自评、直属主管评分、隔级主管复核、HR 校准；一期不实现匿名 360 度互评。
6. 不允许删除已被考核周期引用的指标或方案，只能停用或创建新版本。

### 7.5 考核执行与结果管理

1. 系统按周期适用范围自动生成任务，也支持 HR 对个别员工补发或作废任务。
2. 自评和主管评分均须支持保存草稿、提交、退回修改、附件上传和截止时间控制。
3. 分数由系统根据各评分项权重计算，保留原始分、加权分、总分和等级。
4. 校准只能由授权 HR 执行，须填写调整理由；系统记录调整前后等级、分数和操作人。
5. 结果发布前，员工不可见最终评分；发布后仅可查看本人详情，主管可查看授权团队详情。
6. 员工申诉须填写申诉项、事实说明与附件；复核结论包括维持、调整、撤销，并通知相关人员。
7. 考核归档后只读。需要更正时，应由 HR 发起“归档更正”流程，保留更正版本而非覆盖原数据。

### 7.6 待办、通知与附件

1. 待办中心展示待审批、待填写、待确认、待处理申诉等任务，支持按业务类型和到期时间筛选。
2. 关键节点生成站内消息：申请提交、审批通过/驳回、即将到期、结果发布、申诉结论等。
3. 通知支持已读/未读状态和跳转至业务详情。
4. 附件需记录原始文件名、存储地址、上传人、上传时间、所属业务类型和业务主键；下载前校验业务数据权限。

### 7.7 报表与看板

| 报表 | 核心指标 | 主要使用者 |
| --- | --- | --- |
| 人员结构 | 在职人数、部门人数、岗位/职级分布、司龄、年龄段、性别比例 | 管理层、HR |
| 人员流动 | 入职、转正、调动、离职人数及离职率，按时间和部门统计 | HR、管理层 |
| 假勤统计 | 请假时长、加班时长、余额、审批中数量、异常数量 | HR、部门负责人 |
| 绩效分布 | 各等级人数/比例、平均分、未完成任务、申诉数量 | HR、管理层 |
| 目标完成度 | 部门及个人目标完成率、逾期目标、更新及时率 | 部门负责人、管理层 |

报表应支持按时间范围、部门和岗位过滤，并导出 Excel。对高层默认提供汇总数据，个人明细必须通过权限校验。

## 8. 关键业务规则

1. 已离职员工不能新增请假、加班、目标或绩效任务，但历史记录可按权限查询。
2. 组织、岗位、职级被员工或业务记录引用后不得物理删除，只允许停用。
3. 审批中的业务单据不可直接编辑；申请人可在退回后修改，或按规则撤销后重新发起。
4. 审批链在提交时快照保存。即使后来变更主管，也不影响进行中的单据。
5. 一个员工在同一考核周期仅有一份有效考核任务；补发须先处理原任务。
6. 绩效方案、指标权重、等级规则在周期启动后版本冻结；变更需复制新版本，已生成任务继续使用旧版本。
7. 绩效总分精确到两位小数；等级按已冻结的区间规则计算。边界值归属须在规则中明确。
8. 发布后的绩效结果不得直接修改。任何变更必须通过申诉复核或归档更正流程完成。
9. 所有审批、校准、发布、申诉结论、导入、导出和敏感字段查看均应记录审计日志。
10. 假期余额扣减以“审批通过”为准；撤销或作废后按规则自动返还余额。

## 9. 数据需求与核心实体

| 实体 | 关键字段/关系 |
| --- | --- |
| 用户、角色、权限 | 用户与员工一对一关联；用户可拥有多个角色；角色关联菜单和按钮权限 |
| 部门、岗位、职级 | 部门具备父子层级；员工关联一个当前部门、岗位、职级和直属主管 |
| 员工档案、员工履历 | 档案存当前状态；履历保存每次入转调离的前后快照与流程记录 |
| 审批实例、审批节点 | 关联具体业务单据；保存节点、审批人、动作、意见、时间和快照 |
| 请假规则、余额、申请单 | 规则定义口径；余额按员工和假期类型维护；申请单记录时间段及审批状态 |
| 加班申请、调休余额 | 加班时长、补偿方式、审批结果与调休余额变动关联 |
| 目标周期、目标项 | 目标项可通过 parent_id 形成部门到个人的目标树，并保存进度与证据 |
| 指标库、绩效方案、方案指标 | 方案引用指标及权重，须保留版本号与启用状态 |
| 考核周期、考核任务、评分项 | 任务关联员工、方案和周期；评分项保存自评、主管评分、权重及加权结果 |
| 绩效结果、校准记录、申诉单 | 保存最终分、等级、发布状态、校准前后值、申诉与处理结论 |
| 通知、待办、附件、日志 | 与用户、业务类型、业务主键关联，支持追踪与权限校验 |

所有业务主表建议包含：`id`、`created_by`、`created_time`、`updated_by`、`updated_time`、`deleted`（逻辑删除标识）和乐观锁版本字段。对关键历史表应额外保存业务快照，避免主数据变化导致历史失真。

## 10. 非功能需求

### 10.1 性能与容量

1. 目标规模为 500 至 2,000 名员工、50 个部门、每年 4 个绩效周期。
2. 常规列表查询在正常网络环境下应在 2 秒内返回；普通保存和审批操作应在 3 秒内完成。
3. 月度汇总、导入导出等耗时任务应采用异步任务处理，用户可查看处理进度和失败原因。
4. 对报表采用索引、按时间分区/归档或预聚合策略，避免直接扫描大量明细。

### 10.2 安全与合规

1. 密码必须安全存储，不保存明文；支持密码强度、失效、登录失败限制和首次登录修改密码。
2. 登录会话采用有效期控制与主动失效机制；退出、修改密码和账号禁用后令牌应失效。
3. 身份证号、银行卡、电话等敏感信息需加密存储或脱敏展示，并限制导出。
4. 所有接口必须鉴权，并在服务端执行数据权限校验，前端隐藏菜单不能作为安全边界。
5. 操作审计日志不可由普通业务用户修改；日志保留周期应可配置。

### 10.3 可用性与可维护性

1. 页面应适配常用桌面浏览器，优先支持 1366px 及以上办公分辨率。
2. 提交、审批、发布等关键操作需要二次确认和明确成功/失败反馈。
3. 错误信息面向业务用户应清晰可理解，技术异常写入服务端日志并返回统一错误码。
4. 业务规则、字典、通知模板、审批流配置尽量参数化，减少硬编码。
5. 应提供数据库备份、附件备份和日志归档方案。

## 11. 页面与导航建议

1. 工作台：个人待办、消息、即将到期任务、常用入口、部门概览。
2. 组织人事：组织架构、岗位职级、员工档案、入转调离、员工履历。
3. 假勤管理：请假申请、加班申请、余额管理、审批中心、月度汇总。
4. 目标绩效：目标周期与目标树、指标库、绩效方案、考核周期、考核任务、结果与申诉。
5. 数据分析：人员结构、人员流动、假勤、绩效、目标完成度。
6. 系统管理：用户、角色、菜单、字典、通知、附件、登录日志、操作日志。

## 12. 迭代计划与验收重点

### 12.1 R1：MVP（第一阶段）

1. 完成登录、用户角色、菜单权限和部门数据权限。
2. 完成组织、岗位、职级和员工档案维护。
3. 完成请假申请及直属主管审批。
4. 完成绩效指标、方案、考核周期、员工自评、主管评分、总分计算与结果发布。
5. 完成个人待办、操作日志和基础人员/绩效报表。

### 12.2 R2：完整一期（第二阶段）

1. 入转调离完整流程、离职交接清单。
2. 加班与调休、请假余额自动化规则、月度汇总。
3. 部门到个人的目标分解及进度跟踪。
4. HR 绩效校准、员工申诉、归档更正和消息提醒。
5. Excel 导入导出、附件管理、更多统计看板。

### 12.3 发布范围控制

1. 本文第 4.1 节定义的是完整一期产品边界，不等同于 R1 上线范围；R1 未列出的能力一律在 R2 验收，不得以“需求已存在”为由插入 R1。
2. R1 数据库必须预留 R2 所需的版本、流程、附件与审计字段；R2 通过新增表和可前向兼容的迁移扩展，禁止重写已上线业务历史。
3. 每个需求、接口、数据表、页面和测试用例使用统一编号：`REQ-<域>-<序号>`。例如请假提交为 `REQ-ATT-001`；该编号必须出现在 OpenAPI 的 `operationId` 描述和测试用例中。
4. R1 验收通过条件：全部 R1 必测用例、权限隔离、数据库迁移演练、备份恢复演练和关键 E2E 用例通过。R2 使用相同准入条件，并增加目标、校准、申诉及异步任务恢复用例。

### 12.4 验收标准示例

1. 销售部员工登录后不能查看研发部员工档案和绩效明细；销售部负责人可查看其部门及下级部门数据。
2. 员工请假时，系统能阻止与已通过请假单重叠的时间段，并在审批通过后准确扣减余额。
3. HR 启用绩效周期后，系统为适用员工生成唯一任务；员工自评提交后，任务自动流转至直属主管。
4. 主管评分完成后，系统正确按指标权重计算总分并映射等级；HR 的校准操作有完整前后记录。
5. 绩效结果发布前员工无法查看最终等级；发布后可查看本人结果，并能在截止日期前发起申诉。
6. 员工调岗或离职后，历史请假、目标和绩效记录仍展示当时的组织与岗位快照。

## 13. 技术实现建议（供后续设计使用）

后端可采用 `Spring Boot + Spring Security + MyBatis-Plus + MySQL + Redis + Quartz + MinIO`，前端可采用 `Vue 3 + TypeScript + Element Plus`。系统宜按模块划分组织人事、假勤、目标绩效、系统管理和报表，但一期可保持单体分层架构，避免过早拆分微服务。

审批能力一期可实现可配置的线性审批节点，覆盖直属主管、部门负责人和 HR 等常用场景；复杂条件分支、会签和流程图建模可在后续评估接入 Flowable。绩效计算、余额扣减和状态流转应置于后端事务中，确保数据一致性。


## 14. 前端技术与产品结构（补充）

### 14.1 前端技术选型

前端采用 `Vue 3 + TypeScript + Vite + Vue Router + Pinia + Element Plus + Axios + ECharts`，与 Spring Boot 后端通过 REST API 进行前后端分离协作。

1. `Vue 3` 使用 Composition API 组织业务页面与复用逻辑，适合表单、流程、列表和详情等后台业务场景。
2. `TypeScript` 用于约束前端业务模型和状态，降低权限、审批和绩效状态流转页面的维护风险。
3. `Vite` 提供本地开发、构建、代理和多环境配置能力。
4. `Vue Router` 管理登录、工作台、组织人事、假勤、目标绩效、数据分析和系统管理等路由；路由由后端授权菜单动态生成，并在进入页面前执行权限校验。
5. `Pinia` 管理当前登录用户、角色权限、菜单、待办数量、全局字典和必要的跨页面状态。
6. `Element Plus` 提供表格、表单、树形结构、弹窗、抽屉、上传、日期选择和分页等企业后台基础能力。
7. `Axios` 统一处理认证信息、请求异常、重复提交防护、文件下载和登录失效跳转。
8. `ECharts` 用于人员结构、人员流动、假勤趋势、绩效分布和目标完成度等数据可视化。

前端采用简洁、信息密度适中的企业后台布局：左侧业务导航、顶部用户与通知入口、主工作区面包屑与标签页。复杂业务优先采用“列表 - 详情 - 流程记录”的稳定交互，不在一期引入过度动画或实验性交互。

### 14.2 前端模块边界

| 导航模块 | 面向用户 | 核心目的 |
| --- | --- | --- |
| 工作台 | 全体用户 | 集中展示待办、通知、截止提醒和角色相关摘要。 |
| 组织人事 | HR、部门负责人 | 维护组织与员工信息，处理入转调离并查看员工履历。 |
| 假勤管理 | 员工、主管、HR | 发起和处理请假、加班等申请，并完成汇总管理。 |
| 目标绩效 | 员工、主管、HR、管理层 | 配置考核、分解目标、完成评价、发布结果和处理申诉。 |
| 数据分析 | HR、管理层、授权主管 | 查看人员、假勤、目标和绩效的多维统计。 |
| 系统管理 | 系统管理员、HR | 管理账号、权限、字典、通知模板和审计信息。 |

前端负责呈现业务状态、收集输入、执行基础校验、展示待办与图表，以及基于权限控制可见入口。最终权限判断、审批链生成、绩效计算、余额扣减与敏感信息处理必须由后端再次校验并执行，前端不可作为安全边界。

### 14.3 通用交互要求

1. 列表页支持查询、分页、排序、状态筛选、批量操作和导入导出；可见操作必须与用户权限一致。
2. 详情页应在同一上下文中呈现当前状态、业务信息、流转记录、附件与可执行操作，减少用户跨页面查找。
3. 审批操作应明确展示当前节点、审批历史和操作结果；驳回、退回修改等动作要求填写原因。
4. 发布、归档、离职办结、余额调整和权限调整等高风险操作应提供二次确认和清晰反馈。
5. 页面应完整处理加载、空数据、无权限、网络异常和数据被他人修改等状态，并提示用户下一步操作。
6. 工作台优先展示即将到期和已逾期事项；报表图表与明细共享相同筛选范围，并允许用户在权限范围内下钻查看明细。
