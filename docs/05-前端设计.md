# 05 前端设计

## 1. 技术栈与工程规范

前端使用 Vue 3、TypeScript、Vite、Vue Router、Pinia、Element Plus、Axios 和 ECharts。项目采用按业务域拆分目录的方式，业务页面、API 调用、状态、类型、通用组件和组合式函数各自独立，避免把所有逻辑写入页面组件。

```text
src/
  api/          # 按业务域封装请求
  views/        # 路由页面
  components/   # 通用与业务复用组件
  stores/       # Pinia 状态
  router/       # 静态、动态路由与守卫
  composables/  # 表格、表单、权限等复用逻辑
  types/        # 前端类型定义
  utils/        # 请求、下载、格式化等工具
```

使用 ESLint、Prettier 和 TypeScript 严格模式维持一致代码风格。环境变量区分开发、测试和生产 API 地址；令牌不写入源码或日志。

## 2. 路由与权限

登录后先请求当前用户和授权菜单，再生成动态路由。路由守卫负责未登录跳转、无权限拦截、标题设置和首次加载；按钮权限通过统一指令或组合式函数控制。前端只改善用户体验，后端仍为最终权限校验方。

默认一级导航包括工作台、组织人事、假勤管理、目标绩效、数据分析和系统管理。无权限模块不显示入口，也不得通过手动输入 URL 访问有效内容。

## 3. 页面设计原则

1. 以桌面办公为优先，适配常见 1366px 及以上分辨率；窄屏时保持关键操作可访问。
2. 使用统一的页面骨架：标题与操作区、筛选区、内容区、分页区。避免嵌套卡片和冗余说明。
3. 对有流程状态的对象，详情页固定展示状态、当前处理人、截止时间、操作区和历史记录。
4. 表单采用分组与逐步校验，草稿内容本地保留；提交前显示影响范围和必要确认。
5. 危险操作使用明确文案和二次确认，成功后刷新受影响列表、详情和待办数量。

## 4. 核心页面职责

| 页面区域 | 主要职责 |
| --- | --- |
| 工作台 | 展示个人待办、通知、临期事项和角色相关指标，作为日常入口。 |
| 组织人事 | 组织树与员工目录联动，支持档案查看、异动发起与履历追溯。 |
| 假勤管理 | 提供申请、审批、余额与统计视图，突出时间冲突和待处理异常。 |
| 目标绩效 | 覆盖周期管理、目标进度、方案配置、考核填写、校准、发布和申诉。 |
| 数据分析 | 提供统一筛选条件下的指标、趋势、分布和明细下钻。 |
| 系统管理 | 配置用户、角色、菜单、字典、通知模板与审计查看。 |

## 5. 状态与请求管理

Pinia 中只保存全局或跨页面状态，例如登录身份、授权菜单、通知未读数和字典缓存。页面列表、表单草稿等局部状态应保留在页面或组合式函数中。请求层统一注入令牌、处理 HTTP 错误、刷新会话、转换下载流和避免重复提交。

对列表使用可复用的查询状态，对表单使用显式 loading、saving、submitting 状态。接口返回版本冲突或状态变化时，前端应提示用户刷新详情，不能用旧数据强行覆盖。

## 6. 可视化与无障碍

ECharts 只展示有明确业务意义的趋势和分布，不用图表替代明细。图表必须支持加载、空数据和错误状态，并与筛选条件同步。颜色不能作为唯一状态识别方式，重要状态同时使用文字和图标；表单、弹窗和表格要支持键盘操作与清晰焦点。

## 7. 前端验收重点

1. 不同角色登录后菜单、按钮、数据和待办数量符合权限矩阵。
2. 所有流程页面能正确展示当前状态与历史；重复点击提交不会产生重复业务单据。
3. 所有列表、表单、详情、空状态和异常状态在常用桌面分辨率下可用且无重叠。
4. 文件上传、下载、导入、导出、登录失效和网络错误有一致反馈。
5. E2E 测试覆盖登录、请假审批、绩效自评到发布、权限隔离等关键路径。

## 8. 路由与页面实现清单

路由名称、权限标识和页面职责固定如下。后端返回菜单后，前端只注册已授权的动态路由；详情页再次通过接口鉴权，不能仅依赖路由守卫。

| 路由 | 权限标识 | 页面与关键组件 |
| --- | --- | --- |
| `/dashboard` | `dashboard:view` | 待办、通知、临期事项、角色指标卡；点击进入原业务详情 |
| `/org/departments`、`/org/positions`、`/org/ranks` | `org:read` / `org:manage` | 组织树、基础信息表单、停用校验结果 |
| `/org/employees`、`/org/employees/:id` | `employee:read` / `employee:manage` | 目录、筛选、档案分组、履历、合同、敏感字段脱敏与授权查看 |
| `/personnel/changes`、`/personnel/changes/:id` | `personnel:read` / `personnel:submit` | 单据表单、状态条、审批历史、离职交接清单 |
| `/attendance/leave`、`/attendance/overtime`、`/attendance/balances` | `attendance:read` / `attendance:submit` | 时间段表单、余额提示、个人列表、HR 汇总；不可从页面修改已提交单据 |
| `/goals/cycles`、`/goals/tree` | `goal:read` / `goal:manage` | 周期配置、可展开目标树、进度历史、冻结状态 |

| `/workflow/tasks`、`/workflow/templates` | `workflow:approve` / `workflow:manage` | 审批收件箱、动作弹窗、历史、模板节点与范围配置 |

### 8.1 通用页面状态

每个详情接口响应中的 `availableActions` 是操作区唯一数据来源。通用 `WorkflowActionDialog` 强制显示当前状态、处理对象、审批历史和必填意见；`VersionConflictDialog` 提供刷新和放弃本地更改，不能自动覆盖。上传组件必须在文件为 `AVAILABLE` 后才能把 `fileId` 提交给业务表单。

列表筛选条件必须与 URL query 同步，报表筛选生成不可变快照提交给导出接口。所有导出页显示任务状态、过期时间和失败摘要，下载动作重新请求短时 URL。
