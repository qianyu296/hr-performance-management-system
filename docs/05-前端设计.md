# 05 前端设计

## 1. 技术栈与工程规范

前端使用 Vue 3、TypeScript、Vite、Vue Router、Pinia、Element Plus、Axios 和 ECharts。项目采用按业务域拆分目录的方式，业务页面、API 调用、状态、类型、通用组件和组合式函数各自独立，避免把所有逻辑写入页面组件。

```text
src/
  api/          # 按业务域封装请求
  views/        # 路由页面
  components/   # 通用与业务复用组件
  stores/       # Pinia 状态
  router/       # 路由映射、导航定义与守卫
  composables/  # 表格、表单、权限等复用逻辑
  types/        # 前端类型定义
  utils/        # 请求、下载、格式化等工具
```

项目当前通过 TypeScript 类型检查和 Vite 生产构建保证前端质量，尚未配置 ESLint 或 Prettier。环境变量区分开发、测试和生产 API 地址；令牌不写入源码或日志。

## 2. 路由与权限

路由由 `navigation.ts` 中的静态导航项和 `router/index.ts` 中的组件映射组成。路由守卫负责未登录跳转、首次加载当前用户和权限校验；无权访问导航项时跳转工作台。按钮和操作区同时依据接口返回的可操作标记控制展示，后端仍为最终权限校验方。

默认一级导航包括工作台、组织人事、假勤管理、审批协同、数据分析和系统管理。导航项会按当前用户权限过滤；未知路由及已下架的目标、绩效路由由通配路由返回工作台。

目标与绩效功能采用“前端下架、后端保留”策略：不注册目标或绩效页面路由，不在工作台、报表和流程模板中展示相关入口，也不调用其前端 API。后端接口、数据和现有迁移保留，恢复 UI 前须重新评审。

## 3. 页面设计原则

1. 以桌面办公为优先，适配常见 1366px 及以上分辨率；窄屏时保持关键操作可访问。
2. 使用统一的页面骨架：标题与操作区、筛选区、内容区、分页区。避免嵌套卡片和冗余说明。
3. 对有流程状态的对象，详情页固定展示状态、当前处理人、截止时间、操作区和历史记录。
4. 表单采用分组与逐步校验，草稿内容本地保留；提交前显示影响范围和必要确认。
5. 危险操作使用明确文案和二次确认，成功后刷新受影响列表、详情和待办数量。

## 4. 核心页面职责

| 页面区域 | 主要职责 |
| --- | --- |
| 工作台 | 展示个人待办、通知、临期事项和角色相关指标，作为日常入口。 |
| 组织人事 | 维护部门树、岗位和职级；按部门筛选员工，查看只读任职信息与履历，并发起人事异动。 |
| 假勤管理 | 提供申请、审批、余额与统计视图，突出时间冲突和待处理异常。 |
| 审批协同 | 提供待办任务和请假、加班、人事异动可用的流程模板配置。 |
| 数据分析 | 展示当前启用部门的人员规模与部门人数汇总。 |
| 系统管理 | 配置用户、角色、菜单、字典、通知模板与审计查看。 |

## 5. 状态与请求管理

Pinia 中只保存全局或跨页面状态，例如登录身份、授权菜单、通知未读数和字典缓存。页面列表、表单草稿等局部状态应保留在页面或组合式函数中。请求层统一注入令牌、处理 HTTP 错误、刷新会话、转换下载流和避免重复提交。

对列表使用显式查询、分页、加载和空态状态，对表单使用 `loading`、`saving` 状态避免重复提交。部门维护遇到 `VERSION_CONFLICT` 会提示并刷新部门树及当前节点；其他写操作传递接口返回的版本号，失败时保留服务端错误信息，不使用旧数据强行覆盖。

## 6. 可视化与无障碍

ECharts 只展示有明确业务意义的趋势和分布，不用图表替代明细。图表必须支持加载、空数据和错误状态，并与筛选条件同步。颜色不能作为唯一状态识别方式，重要状态同时使用文字和图标；表单、弹窗和表格要支持键盘操作与清晰焦点。

## 7. 前端验收重点

1. 不同角色登录后菜单、按钮、数据和待办数量符合权限矩阵。
2. 所有流程页面能正确展示当前状态与历史；重复点击提交不会产生重复业务单据。
3. 所有列表、表单、详情、空状态和异常状态在常用桌面分辨率下可用且无重叠。
4. 登录失效、网络错误、校验错误和版本冲突有明确且可恢复的反馈。
5. 以 `npm.cmd run typecheck` 和 `npm.cmd run build` 验证类型与生产构建；关键角色和人事流程以人工验收覆盖。

## 8. 路由与页面实现清单

导航路由来自 `navigationItems`，详情路由单独静态注册。守卫只负责前端访问体验，详情和所有写操作仍由后端接口再次鉴权。

| 路由 | 权限标识 | 页面与关键组件 |
| --- | --- | --- |
| `/dashboard` | 已登录 | 工作台与各业务入口 |
| `/org/departments` | `org:read`；维护动作要求 `org:manage` | 部门树、负责人、排序、生效信息；支持新建、编辑、移动、停用和冲突后的刷新 |
| `/org/employees` | `org:read`；普通资料更新及岗位职级维护要求 `org:manage` | 员工目录、部门/状态筛选、岗位职级弹窗；档案抽屉的当前任职只读，显示履历并可发起异动 |
| `/personnel/changes` | `personnel:read` | 异动列表，按当前部门、员工、类型、状态和生效日期筛选；草稿可创建或编辑 |
| `/personnel/changes/:id` | `personnel:read` | 异动前后快照对比、草稿编辑、提交、撤回、受控生效和离职交接事项 |
| `/attendance/leave`、`/attendance/leave-types`、`/attendance/overtime`、`/attendance/calendar`、`/attendance/balances`、`/attendance/summary` | 对应 `attendance:*` 权限 | 请假、加班、日历、余额、类型和月度汇总 |
| `/workflow/tasks`、`/workflow/templates` | `workflow:approve` / `workflow:manage` | 审批收件箱、动作弹窗、历史、模板节点与范围配置 |
| `/reports/overview` | `report:read` | 当前启用部门的人员规模与部门人数汇总 |
| `/system/users` | `system:manage` | 用户、角色、菜单和审计配置 |

`/goals/*`、`/performance/*` 不在路由与组件映射中；直接访问会命中通配路由并返回 `/dashboard`。

### 8.1 通用页面状态

人事异动详情响应中的 `canEdit`、`canSubmit`、`canWithdraw`、`canMaintainHandover`、`canExecute` 是操作区唯一数据来源；离职交接事项使用各自的 `canConfirm` 控制确认按钮。创建草稿不携带版本；更新、提交、撤回、生效和交接确认携带字符串形式的 `version`，成功响应替换当前详情，避免继续使用旧版本。

员工档案编辑仅提交姓名、性别和版本；部门、岗位、职级、主管、任职状态与日期由人事异动处理。离职详情仅在离职类型显示交接清单，具备维护权限者可新增事项，接收人或具备管理权限者可确认事项。当前列表筛选保留在页面查询状态中，尚未与 URL query 或异步导出任务集成。
