# 06 权限与审批设计

## 1. 设计目标

权限体系同时解决“谁能进入功能”和“谁能查看或处理哪些数据”。审批体系解决“业务何时可生效、由谁处理、过程如何追溯”。两者均由服务端强制执行，前端仅按授权结果展示入口与操作。

## 2. 权限模型

采用 RBAC 加数据范围模型：用户关联员工，用户可拥有多个角色；角色关联菜单、按钮和数据范围。最终权限取并集，但敏感字段与高风险动作可以采用更严格的专项授权。

| 角色 | 功能边界 | 数据范围 |
| --- | --- | --- |
| `SUPER_ADMIN` | 系统配置、组织人事、假勤、审批与日志管理 | 初始化为 `ALL` 数据范围；拥有受控的人事生效权限 |
| `HR_SPECIALIST` | 组织员工、人事异动草稿/交接维护与审批任务处理 | 由角色数据范围约束；种子数据不默认授予 `personnel:execute` |
| `EMPLOYEE_SELF_SERVICE` | 个人请假、加班等自助操作 | 本人及接口规则允许的数据 |
| 部门主管/普通员工 | 由业务角色、数据范围和审批任务分配决定 | 不预置独立角色；接口必须校验数据范围和任务归属 |

## 3. 数据范围规则

系统支持本人、直属下属、本部门、本部门及下级部门、全公司和自定义组织范围。数据范围应转换为后端查询条件，不把完整数据返回给前端后再过滤。员工从部门调出后，新权限只影响后续实时查询；已发起流程的审批人和组织关系按提交时快照处理。

敏感信息执行字段级控制：默认脱敏展示，只有具备专项权限的用户可查看完整内容。敏感查看、批量导出和文件下载应接入审计日志；当前已接入的审计事件见第 6 节。

## 4. 审批模板

一期使用线性审批模板，节点类型包括直属主管、部门负责人、HR、指定人员和发起人自选审批人。模板按业务类型、组织范围和启停状态配置。提交时根据有效模板生成审批实例与节点快照；模板后续调整不影响已在途单据。

| 业务类型 | 推荐节点 | 最终生效动作 |
| --- | --- | --- |
| 入职/转正 | 由 `PERSONNEL_CHANGE` 模板节点配置 | 已到生效日时创建或变更员工任职状态 |
| 调动/晋降/停职/离职 | 由 `PERSONNEL_CHANGE` 模板节点配置 | 写入履历并更新当前状态；离职还需完成必办交接 |
| 请假/加班 | 由对应模板节点配置 | 更新申请状态和余额/调休流水 |

## 5. 流程状态与动作

通用审批状态为草稿、审批中、已通过、已驳回、已撤销、已失效。审批节点只允许待处理、已通过、已驳回、已转交等受控状态。发起人可在规则允许时撤回；审批人只能处理分配给自己的有效任务；管理员不得绕过流程直接将业务置为通过。

驳回终止当前流程，退回修改则返回发起人并保留历史意见。转交、委托、加签等高级能力不在一期范围，但数据模型与动作日志应保留扩展位置。

### 5.1 已实现的人事异动授权与状态规则

人事异动以 `PERSONNEL_CHANGE` 业务类型接入通用工作流。接口首先校验权限，再由应用服务校验员工组织数据范围、单据创建人、接收人和版本号；页面只根据服务端返回的 `canEdit`、`canSubmit`、`canWithdraw`、`canMaintainHandover`、`canExecute` 与交接事项的 `canConfirm` 展示操作，不能替代服务端判断。

| 权限 | 服务端允许的动作 | 附加约束 |
| --- | --- | --- |
| `personnel:read` | 查询异动列表、查看详情和履历 | 已有关联员工按员工可读数据范围校验；尚未生效的入职单仅创建人或 `personnel:manage` 可查看 |
| `personnel:create` | 创建、修改草稿、提交、撤回、维护离职交接事项、确认交接事项 | 创建、修改、提交和交接维护还要求创建人或 `personnel:manage`，以及相关员工/部门可写；撤回仅创建人可执行 |
| `personnel:manage` | 在授权组织范围内管理他人异动和确认任意交接事项 | 仍受员工与部门可写数据范围限制，不替代流程审批任务归属 |
| `personnel:approve` | 在审批中心处理分配给本人的 `PERSONNEL_CHANGE` 工作流任务 | 审批任务、模板版本、审批人和节点快照均以提交时数据为准 |
| `personnel:execute` | 对已通过且已到生效日的异动执行生效 | 仅用于受控生效动作；当前种子授权仅赋予 `SUPER_ADMIN`，HR 专员不默认拥有 |

当前状态机为：`DRAFT -> IN_PROGRESS -> APPROVED -> EFFECTIVE`，并支持 `IN_PROGRESS -> REJECTED`、`IN_PROGRESS -> WITHDRAWN` 和工作流退回后的 `IN_PROGRESS -> DRAFT`。创建和编辑时保存异动前后任职快照；提交时选择目标部门匹配的 `PERSONNEL_CHANGE` 模板并冻结流程节点。最终审批在生效日不晚于当前日期时自动执行生效；未来日期停留在 `APPROVED`，只能由具备 `personnel:execute` 的用户在日期到达后执行。

最终审批会先置单据为 `APPROVED`：生效日不晚于当前日期时，在同一审批事务内自动执行生效；未来生效日则等待具备 `personnel:execute` 的用户执行受版本保护的生效动作。生效事务更新或创建员工、追加不可变履历、处理入职账号开通或离职账号禁用、更新异动状态并记录审计。修改单据、状态动作和交接确认均携带版本；受影响行数不符合预期时返回冲突，不覆盖较新的数据。新增交接事项不需要交接单版本。

### 5.2 离职交接规则

创建离职异动时系统同步建立离职交接单。创建人或具备 `personnel:manage` 的用户仅能在 `DRAFT`、`IN_PROGRESS` 阶段新增事项；接收人必须是有效且在授权范围内的员工。事项可标记为必办，状态从 `PENDING` 变为 `CONFIRMED`。

交接确认接口要求 `personnel:create`，并额外要求操作者为该事项的接收人；具备 `personnel:manage` 的用户可以代为确认。服务端将此结果返回为每项 `canConfirm`，前端不向无权用户展示确认按钮。任一必办事项仍为待确认时，离职异动不得生效；全部必办事项确认后交接单标记为完成。离职生效后才禁用关联账号并使旧会话版本失效，历史流程与人事记录不删除。

## 6. 审计与异常处理

当前 `OperationAuditService` 只记录人事异动生效；登录、权限变更、敏感查看、导入导出、审批、离职交接和余额调整的审计属于待接入范围。审批人离职、调岗或长时间未处理时，当前仅允许具有 `workflow:intervene` 的用户转交任务；改派、终止和 `BLOCKED` 状态尚未实现。流程模板缺失、审批人不存在或数据范围不足时，提交失败且不创建半成品流程。

## 7. 验收要求

1. 不同角色获得正确菜单、按钮及数据范围，且后端接口无法越权。
2. 提交审批后节点、审批人和历史记录清晰可追溯；变更组织后在途流程仍按快照处理。
3. 敏感信息默认脱敏，查看完整内容、导出和下载均受授权并被记录。
4. 非法状态操作、重复审批、越权审批、已归档对象修改均被服务端拒绝。

## 8. 权限判定算法与授权矩阵

权限判定顺序固定为：身份有效 -> 菜单/按钮权限 -> 字段策略 -> 数据范围 -> 当前对象状态 -> 流程任务归属。任一步失败即拒绝，不能因用户具有更大数据范围而绕过流程任务归属。

1. 多角色的菜单、按钮和常规数据范围按并集计算；`ALL` 范围覆盖其他常规范围。
2. 敏感字段、导出、完整下载、余额调整、人员异动生效和权限变更均要求显式功能授权，不能仅因 `ALL` 数据范围获得。
3. 自定义范围必须由角色关联 `sys_role_data_scope`，再由 `sys_data_scope_dept` 枚举部门；不得把部门 ID 列表从前端传入查询。
4. `SELF` 比较当前用户关联的 employee ID；`DIRECT` 比较当前有效直属主管；`DEPT_TREE` 使用部门物化路径或递归查询生成。流程和历史详情先以业务快照校验，再读取关联对象。

| 资源/动作 | 普通员工 | 直属主管 | 部门负责人 | HR 专员 | HR 负责人 | 系统管理员 | 财务/高层 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 员工档案 | 本人有限字段 | 直属团队 | 部门树 | 全公司 | 全公司 | 仅系统字段 | 授权只读/聚合 |
| 人事异动 | 无预置权限 | 审批被分配任务 | 审批被分配任务 | 草稿、交接维护及审批任务 | 由角色授权决定 | 全部受控动作 | 无 |
| 请假、加班、余额 | 本人申请/查看 | 审批直属任务 | 授权范围查看 | 全公司管理；专项调整 | 全公司查看 | 无 | 授权只读 |
| 权限、菜单、审计 | 无 | 无 | 无 | 仅审计查询授权 | 审计查询授权 | 管理（敏感明细另授） | 无 |

## 9. 审批模板选择与异常规则

模板选择输入为业务类型、发起人部门快照、申请金额/时长等业务上下文和提交时刻。筛选启用且生效的模板后，按“组织范围最具体、优先级最高、版本最高”选择；若仍有多条匹配，提交失败并记录配置异常，禁止随机选择。

当前审批人解析按模板节点快照创建任务，未实现连续节点去重或 `self_approval_policy`。`BLOCKED`、改派和终止流程同样未实现；需要人工干预时，仅具备 `workflow:intervene` 的用户可调用转交接口，并记录转交原因和动作日志。

线性流程状态：`DRAFT -> IN_PROGRESS -> APPROVED | REJECTED | WITHDRAWN | INVALID`；业务单据可有独立的 `EFFECTIVE` 状态。`RETURNED` 不是流程终态：保留实例和全部历史意见，业务单据可编辑；重新提交复用该实例、创建新的待处理节点并写入 `RESUBMIT` 动作日志。所有流程快照至少包含模板版本、节点规则、审批人、发起人部门和主管链。

## 10. 当前产品范围

当前交付定位为“人力资源管理系统”。目标与绩效相关的后端接口、数据表、迁移和历史记录保留，但 UI 已冻结：不注册原目标/绩效路由，不在侧边栏、工作台、数据分析页和流程模板页展示对应入口或数据；直接访问原前端地址由通配路由返回工作台。`PERFORMANCE_APPEAL` 历史流程模板保留在数据库中，但不在模板列表或新建类型中显示。恢复该 UI 需要独立评审，不得通过单独恢复菜单或页面绕过冻结策略。
